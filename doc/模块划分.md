# PaaS 平台模块划分文档

## 1. 目标

本文档用于指导 PaaS 平台按模块拆分开发，使各模块具备清晰边界、低耦合、可独立开发、可独立测试和可逐步演进的能力。

模块划分遵循 `doc/需求.md` 和 `doc/总体设计.md` 中的产品定位、领域模型和架构约束。

核心目标：

- 每个模块只负责一个清晰的业务能力。
- 模块之间不直接访问彼此内部数据结构。
- 模块之间通过接口、事件或明确的数据契约协作。
- 每个模块可以用 mock、fake adapter 或测试桩独立完成单元测试和集成测试。
- 当前测试可用版本采用模块化单体架构，保留未来拆分为独立服务的边界。

## 2. 划分原则

### 2.1 按领域能力拆分

模块边界优先按业务领域划分，而不是按技术层划分。每个模块内部可以包含自己的 domain、service、repository、handler 和 adapter。

推荐模块内部结构：

```text
module-name/
  domain/       # 领域对象、枚举、状态机、领域规则
  application/  # 用例编排、事务边界、权限前置检查
  ports/        # 模块依赖的接口定义
  adapters/     # 外部系统或基础设施适配
  api/          # HTTP/RPC handler、DTO、路由注册
  repository/   # 本模块数据访问实现
  tests/        # 模块测试
```

### 2.2 模块不共享数据库访问

每个模块只能通过自己的 repository 访问本模块拥有的数据表。其他模块如需读取信息，应通过模块公开的 query service、application service 或事件投影获取。

禁止：

```text
build 模块直接查询 applications 表
delivery 模块直接修改 build_runs 表
agent 模块直接修改 environments 表之外的其他模块状态
```

允许：

```text
build 模块通过 ApplicationQueryPort 获取 ApplicationSource
delivery 模块通过 BuildQueryPort 获取可发布 BuildArtifact
agent 模块通过 ClusterCommandPort 更新集群状态
```

### 2.3 外部系统通过适配器隔离

GitLab、Jenkins、Argo CD、Kubernetes 不应出现在核心领域逻辑中。核心模块只依赖 port interface，具体调用放在 adapter 模块中。

示例：

```text
SourceRepositoryService -> GitRepositoryPort -> GitLabAdapter
BuildService            -> BuildRunnerPort    -> JenkinsAdapter
DeliveryService         -> ManifestPort       -> GitLabManifestAdapter
AgentGateway            -> ClusterTaskPort    -> AgentTaskStore
```

### 2.4 模块间协作优先使用事件

跨模块的后续动作优先通过领域事件触发，避免同步调用链过长。

示例：

```text
BuildSucceeded
  -> release 模块生成 Release
  -> delivery 模块生成 Freight

PromotionApproved
  -> gitops 模块修改部署清单
  -> deployment 模块创建 Deployment

AgentStatusReported
  -> cluster 模块更新 Cluster 状态
  -> environment 模块更新 EnvironmentState
```

## 3. 总体模块视图

```text
paas
  |-- web-console
  |-- api-server
  |     |-- identity-access
  |     |-- tenant-project
  |     |-- source-repository
  |     |-- application-environment
  |     |-- build
  |     |-- release-delivery
  |     |-- gitops-deployment
  |     |-- cluster-agent
  |     |-- audit
  |     `-- notification
  |-- paas-agent
  |-- integrations
  |     |-- gitlab
  |     |-- jenkins
  |     |-- argocd
  |     `-- registry
  `-- shared-kernel
```

模块分为五类：

- 前端模块：`web-console`
- 控制面业务模块：`identity-access`、`tenant-project`、`source-repository`、`application-environment`、`build`、`release-delivery`、`gitops-deployment`、`cluster-agent`、`audit`、`notification`
- 集群侧模块：`paas-agent`
- 外部系统适配模块：`integrations`
- 最小共享内核：`shared-kernel`

## 4. 模块清单

### 4.1 web-console

职责：

- 提供 PaaS Web 控制台。
- 实现登录、项目列表、应用列表、创建应用向导、应用详情、环境详情、构建详情、发布晋级、配置、审计日志等页面。
- 所有用户可见文案必须使用中文，包括菜单、按钮、表单标签、校验提示、状态、空状态和错误提示。
- 登录页必须同时支持本地账号密码登录和 OIDC 登录入口。
- 通过 API Server 访问后端能力。

推荐技术栈：

```text
React
TypeScript
Vite
Ant Design
React Router
TanStack Query
Zustand
```

技术栈说明：

- React 用于构建复杂控制台页面和交互组件。
- TypeScript 用于约束 API DTO、状态枚举、权限点和领域模型。
- Vite 用于前端开发、构建和本地调试。
- Ant Design 用于后台系统常见的表格、表单、步骤条、抽屉、标签和审批交互。
- React Router 用于页面路由。
- TanStack Query 用于 API 请求、缓存、加载状态和错误状态管理。
- Zustand 用于少量全局状态，例如当前用户、当前租户、当前项目和前端偏好设置。

不负责：

- 不直接访问 GitLab、Jenkins、Argo CD 或 Kubernetes。
- 不保存后端业务状态。
- 不实现权限判定的最终逻辑。

依赖：

- `api-server` 提供的 HTTP API。

独立测试：

- 使用 mock API 测试页面状态。
- 使用组件测试覆盖创建应用、构建日志、发布审批等交互。
- 使用端到端测试覆盖核心用户路径。

### 4.2 identity-access

- 用户、身份、角色、权限、RoleBinding、Group、ServiceAccount、AccessToken 管理。
- 支持平台自主创建本地用户，并提供本地账号密码登录。
- 支持 OIDC Provider 配置、OIDC 登录回调和 OIDC Identity 到 User 的映射。
- RBAC 权限判定。
- 登录态和访问令牌校验。
- 为其他模块提供权限检查能力。

核心对象：

```text

```

```text

role_bindings
service_accounts
access_tokens
```

对外接口：


不负责：

- 不管理业务资源生命周期。
- 不同步 GitLab 权限，GitLab 权限同步由 `source-repository` 发起。

独立测试：

- 测试 RoleBinding 作用域继承。
- 测试 User、Group、ServiceAccount 权限合并。
- 测试生产发布审批相关权限点。

## 5. 模块依赖关系

推荐依赖方向：

```text
web-console
  -> api-server

api-server modules
  -> shared-kernel
  -> module ports
  -> integrations adapters

paas-agent
  -> Kubernetes API / Argo CD CRD
  -> cluster-agent Agent Gateway API
```

业务模块之间允许依赖对方公开的 port interface，不允许依赖对方内部实现。

推荐依赖图：

```text
identity-access  <- all modules
tenant-project   <- source-repository, application-environment, audit
source-repository <- application-environment, build
application-environment <- build, release-delivery, gitops-deployment, cluster-agent
build <- release-delivery
release-delivery -> gitops-deployment
cluster-agent -> application-environment, gitops-deployment
audit <- all modules
notification <- event bus
```

模块间推荐通过以下事件解耦：

```text

DeploymentStarted

```

事件使用要求：

- 事件必须包含 event_id、event_type、occurred_at。
- 事件 payload 只传必要 ID 和稳定字段。
- 消费方必须保证幂等。
- 事件失败应可重试。

## 7. API 边界

API Server 对前端暴露统一 API，但内部路由归属到不同模块。

建议 API 分组：

```text
/api/auth/*
/api/tenants/*
/api/projects/*
/api/source-repositories/*
/api/repository-migrations/*
/api/apps/*
/api/environments/*
/api/builds/*
/api/releases/*
/api/freights/*
/api/promotions/*
/api/deployments/*
/api/clusters/*
/api/audit-logs/*
/api/agent/v1/*
```

路由归属：


## 8. 数据边界

每个模块拥有自己的表和迁移文件。其他模块不得直接写入不属于自己的表。

数据库实现约束：

- PaaS 默认数据库为 MySQL 8.0+。
- 所有表默认使用 InnoDB。
- 所有字符字段默认使用 utf8mb4 字符集。
- 迁移 SQL 以 MySQL 8.0 方言为准。
- 业务仓储必须落在 MySQL 正式表，不得使用内存仓储、内存快照或 `repository_snapshots` 快照表作为持久化方案。
- 不使用 PostgreSQL 专属语法或数据类型，例如 `RETURNING`、`jsonb`、原生 `uuid` 类型、数组列。
- JSON 字段可以使用 MySQL `JSON` 类型，但不得把复杂检索、全文检索或日志分析压在 JSON 查询上。
- repository 测试必须覆盖事务、唯一索引、外键、分页和状态更新。

## 10. 独立开发和测试策略

每个模块至少提供三类测试能力：

- 单元测试：测试领域规则、状态机、权限前置判断和错误处理。
- 模块集成测试：使用 fake adapter 或测试数据库验证模块用例。
- 契约测试：验证模块对外 port、事件和 API 的输入输出稳定性。

开发要求：

- 模块开发时不得要求真实 GitLab、Jenkins、Argo CD 或 Kubernetes 必须可用。
- 外部系统依赖必须有 fake 或 mock 实现。
- 关键状态机必须有表驱动测试。
- 跨模块事件消费必须可重复执行，保证幂等。
- API DTO 不直接暴露数据库模型。

## 11. 测试可用版本开发顺序建议

建议按以下顺序开发，减少跨模块阻塞：

```text
1. shared-kernel、platform 基础设施
2. identity-access
3. tenant-project
4. source-repository
5. application-environment
6. build
7. release-delivery
8. gitops-deployment
9. cluster-agent
10. paas-agent
11. web-console
12. audit、notification 横向补齐
```

也可以并行开发：

```text
identity-access + tenant-project
source-repository + integrations/gitlab
build + integrations/jenkins
cluster-agent + paas-agent
web-console + mock API
```

并行开发前需要先稳定：

- 核心 ID 命名。
- API DTO。
- 事件名称和 payload。
- 权限点命名。
- 关键状态枚举。

## 12. 验收标准

模块划分达到以下标准即可进入编码阶段：

- 每个模块有明确职责和非职责。
- 每个模块有清晰拥有的数据表。
- 模块间依赖只通过接口、事件或 API。
- 每个模块可以用 fake adapter 独立测试。
- 外部系统调用均封装在 `integrations` 或 Agent 内部。
- 控制面不直接调用 Kubernetes API Server。
- Agent 不执行业务发布决策。
- Application、Environment、SourceRepository、Release、Freight、Promotion 等核心概念与项目文档保持一致。
