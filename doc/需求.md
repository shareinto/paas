# PaaS 平台设计备忘

本文档汇总本次会话中讨论的 PaaS 平台设计。

## 产品定位

建设企业内部 PaaS 平台，用户只通过 PaaS Web 控制台操作。Jenkins、GitLab 部署清单仓库、Argo CD 和 Kubernetes API 对普通用户隐藏。

核心集成系统：

- GitLab：用于源码仓库和部署清单仓库。
- Jenkins：用于构建。
- Argo CD：用于 GitOps 部署。
- Kubernetes：用于运行时。
- PaaS Agent：部署在每个 Kubernetes 集群内，用于状态上报和受控的集群侧任务。

## 关键架构决策

- PaaS 控制面是唯一面向用户的控制入口。
- PaaS 不直接调用 Kubernetes API Server。
- 每个 Kubernetes 集群都有自己的 Argo CD 实例。
- 每个 Kubernetes 集群运行一个 PaaS Agent。
- PaaS Agent 在集群内读取 Argo CD Application 和工作负载状态，然后主动上报给 PaaS 控制面。
- 用户看不到 Jenkins UI、GitLab 部署清单仓库、Argo CD UI 或 kubeconfig。
- 源码仓库默认由平台托管在 GitLab 中。
- 现有源码仓库可以迁移到平台托管仓库，但不直接作为运行时源码使用。
- 创建 Application 时不创建源码仓库。用户必须在创建 Application 时指定一个已有的平台托管 SourceRepository。
- Application 表示最小的、可独立构建、部署、扩缩容和回滚的交付单元。
- Application 默认不表示完整业务系统。前端、后端服务、worker 或微服务通常应建模为独立的 Application。
- 当前测试可用版本默认每个 Application 产出一个主镜像，同时在 Release/Freight 和构建产物模型上保留多产物扩展能力。
- 一个 SourceRepository 可以被多个 Application 复用，因此同时支持单仓库和 monorepo 源码布局。
- Web Console 所有用户可见文案必须使用中文，不使用英文作为页面标题、按钮、菜单、提示或状态文案。
- 用户登录支持两种途径：企业 OIDC 登录，以及平台内自主创建的本地用户登录。

## Environment、Stage 和 Cluster 的关系

Environment 是运行时上下文。Stage 是交付流节点。Cluster 是实际运行资源池。

关系：

```text
Application
  ├── Environment
  │     └── EnvironmentClusterBinding
  │           └── Cluster
  └── DeliveryFlow
        └── DeliveryStage
              └── Environment
```

不要使用 Placement 术语。使用：

- EnvironmentClusterBinding
- environment_cluster_bindings
- ClusterBindingPolicy
- pending_cluster_binding
- cluster_bound

## Application 和仓库粒度

Application 是平台的交付单元：

```text
Application = 一个可独立构建/部署的组件
当前测试可用版本输出 = 一个主容器镜像
```

示例：

```text
Project: payment
  ├── Application: user-api
  ├── Application: order-api
  └── Application: admin-web
```

如果用户描述的是由前端加后端服务组成的更大系统，应将其建模为同一 Project 下的多个 Application。未来可以增加 ApplicationGroup/System 概念，用于导航和协同发布，但不应替代 Application 作为交付单元的语义。

仓库策略：

```text
Single repo:
  SourceRepository user-service -> Application user-service

Monorepo:
  SourceRepository commerce-platform
    ├── Application user-api   source_path=services/user-api
    ├── Application order-api  source_path=services/order-api
    └── Application admin-web  source_path=frontend/admin-web
```

规则：

- 一个 SourceRepository 可以支撑多个 Application。
- Application 可以包含多个 ApplicationSource。每个 ApplicationSource 保存一个代码源的标识、仓库、源码路径、默认 ref 和构建规格 BuildSpec。当前测试可用版本的 BuildSpec 支持 Java 构建，包含 `build_command`、`artifact_copy_command`、`runtime_base_image`、`artifact_deploy_path` 和 `default_ref`。
- 创建 Application 时，运行时环境按应用配置一次，包括 `runtime_base_image` 和 `artifact_deploy_path`。多代码源应用不会为每个代码源重复展示运行时环境配置。
- 一个 Build 可以检出多个代码源，并从多个代码源生成多个 BuildArtifact；默认 `is_primary=true` 的产物用于列表和兼容单镜像部署展示。
- Release 属于一个 Application。
- Freight 使用 FreightItem，因此未来可以组合多个 Application 的 Release 做协同晋级。
- 默认不要让一个 Application 表示多个可独立部署的微服务。
- 当 sidecar 或强耦合辅助容器与主工作负载生命周期不可分离时，可以归属于同一个 Application。

## 默认环境行为

创建应用时，PaaS 应根据环境模板创建默认环境。

当前测试可用版本默认环境：

```text
dev
test
staging
prod
```

如果尚未注册任何集群：

- 仍然创建环境。
- 环境状态为 `pending_cluster_binding`。
- 不创建 EnvironmentClusterBinding。
- 不创建 Argo CD Application。
- 构建可以运行。
- 部署会被阻止，直到存在集群绑定。

集群注册后：

```text
1. 管理员注册 Cluster。
2. PaaS 扫描 pending_cluster_binding 环境。
3. PaaS 手动或通过 ClusterBindingPolicy 创建 EnvironmentClusterBinding。
4. PaaS 创建 namespace、部署清单路径和 Argo CD Application。
5. Environment 变为可部署。
```

## GitLab 策略

使用平台托管源码仓库。

映射关系：

```text
Tenant      -> GitLab Group
Project     -> GitLab Subgroup
SourceRepository -> GitLab Project
```

推荐 GitLab 目录树：

```text
paas-managed
  └── tenant-a
        └── payment
              └── user-service
```

PaaS 角色到 GitLab 角色的映射：

```text
tenant_owner  -> Owner
project_admin -> Maintainer
developer     -> Developer
viewer        -> Reporter
```

GitLab 权限只表示源码权限。构建、部署和生产审批仍由 PaaS 控制。

SourceRepository 创建和迁移是独立流程。Application 创建时通过 `sources` 数组选择一个或多个已有 SourceRepository，并为每个代码源创建带有源码路径、默认 ref 和 BuildSpec 的 ApplicationSource 绑定。对于单仓应用，通常只有一个代码源。对于多仓构建场景，一个 Application 可以同时绑定多个代码源。对于 monorepo 应用，多个 Application 仍可以通过不同的 ApplicationSource 路径引用同一个 SourceRepository。

## Jenkins 构建配置

Jenkins 对用户隐藏。

PaaS 创建并管理 Jenkins Job。Application 创建时不创建 Jenkins Job；用户点击构建时，PaaS 使用固定名称确保对应 Jenkins Job：

```text
paas/{tenant_name}/{project_name}/{app_name}
```

PaaS 每次触发构建前都会按当前构建配置渲染 Jenkins Job。Job 不存在时创建，已存在时更新配置，然后触发 Jenkins 构建；不再计算 md5 指纹，也不再通过 md5 后缀创建新 Job。

当前测试可用版本由 PaaS 在创建或编辑 Application 时保存 BuildSpec，不要求 Jenkins 自己适配所有源码结构；Jenkins 只执行平台统一模板。

BuildSpec 字段：

```text
source_path
build_command
artifact_copy_command
runtime_base_image
artifact_deploy_path
default_ref
```

约束：

- Java 构建由用户填写的 BuildSpec 与构建环境镜像表达，不再在 BuildSpec 中保存构建类型或语言版本字段。
- 用户填写构建命令，但命令只允许在 `source_path` 下执行。
- 镜像生成统一使用平台维护的运行时基础镜像，`runtime_base_image` 必须来自平台允许列表。
- `artifact_deploy_path` 是可选产物放置路径；填写时必须是绝对路径，且不允许包含 `..`。
- 在创建应用页面，`runtime_base_image` 和 `artifact_deploy_path` 是应用级运行时环境配置，不随代码源数量变化。
- `dockerfile_path` 是运行时环境管理配置，只在平台管理员的运行时环境管理页维护；创建应用和触发构建时不向用户展示。
- 当前测试可用版本暂不支持客户仓库自带 Jenkinsfile。
- 当前测试可用版本不要求客户在源码仓库提供 Dockerfile；Dockerfile 由平台统一维护在平台专用 Git 仓库中。Jenkins 构建时检出平台 Dockerfile 仓库，Spring Boot jar 默认使用 `java/jar/Dockerfile`，Tomcat war 使用 `java/tomcat/Dockerfile`，并通过 `docker buildx` 构建多架构镜像。

构建日志：

- PaaS 使用 Jenkins progressiveText API。
- 前端通过 SSE 或 WebSocket 从 PaaS 流式接收日志。
- 用户只在 PaaS 中查看日志。

完整构建适配规则见 `doc/Java构建适配方案.md`。

## GitOps 部署策略

PaaS 更新 GitLab 部署清单仓库来表达期望状态。

Argo CD 从 Git 同步到 Kubernetes。

回滚同样通过修改 Git 中的期望状态完成，而不是直接变更集群。

平台提供默认应用部署模板，但共享基础模板只读。不同租户、项目或 Application 如需自定义 initContainer、目录创建、目录权限、volumeMount、安全上下文等部署细节，应通过 PaaS 创建自己的部署模板副本或 overlay。用户不直接访问 GitLab 部署清单仓库，模板变更由 PaaS 校验、审计并写入清单仓库。

## PaaS Agent

Agent 职责：

- 心跳。
- 集群资源快照。
- 监听 Argo CD Application。
- 监听 Deployment/StatefulSet/ReplicaSet/Pod/Event。
- 向 PaaS 上报标准化后的应用状态。
- 执行受控任务，例如 Argo CD refresh/sync。

Agent API：

```text
POST /api/agent/v1/heartbeat
POST /api/agent/v1/status/report
POST /api/agent/v1/events/report
GET  /api/agent/v1/tasks
POST /api/agent/v1/tasks/result
```

## 前端 UX

前端展示要求：

- 所有用户可见文案必须使用中文，包括导航、按钮、表单标签、校验提示、状态、空状态、错误提示和确认弹窗。
- 技术枚举、API 字段、日志原文和代码片段可以保留英文，例如 `java_springboot`、`build_command`、`source_path`。
- 登录页必须支持本地账号密码登录和 OIDC 登录两种入口。

主要用户概念：

- Application
- Environment
- Version/Freight
- Build
- Promotion
- Deployment status

测试可用版本页面：

- 登录
