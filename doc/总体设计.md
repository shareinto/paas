# PaaS 平台实现文档

## 1. 建设目标

本平台定位为企业内部应用交付 PaaS。用户只登录 PaaS 平台，即可完成应用创建、源码托管、构建、环境配置、版本晋级、发布审批、运行状态查看、日志查看和回滚。

平台底层集成 GitLab、Jenkins、Argo CD 和 Kubernetes，但这些系统不直接暴露给普通用户：

- GitLab：承载源码仓库和部署清单仓库。
- Jenkins：执行构建流水线。
- Argo CD：基于 GitOps 将部署清单同步到 Kubernetes。
- Kubernetes：应用运行面。
- PaaS Agent：部署在每个 Kubernetes 集群中，负责状态采集和受控任务执行。

核心原则：

- PaaS 是统一入口、权限中心和业务编排中心。
- GitLab 是源码和部署期望状态的来源。
- Jenkins 只负责构建，不作为用户入口。
- Argo CD 只负责部署同步，不作为用户入口。
- PaaS 控制面不直连 Kubernetes API Server。
- 每个集群内的 PaaS Agent 主动连接 PaaS 控制面，上报状态。

## 2. 总体架构

```text
User Browser
  |
  v
PaaS Web Console
  |
  v
PaaS Control Plane
  |
  |-- PaaS Database (MySQL 8.0+)
  |-- GitLab Adapter
  |     |-- Source Repository API
  |     `-- Manifest Repository API
  |-- Jenkins Adapter
  |     |-- Job Management
  |     |-- Build Trigger
  |     `-- Progressive Build Log
  |-- Argo/GitOps Adapter
  |     `-- Manifest Change / MR / Commit
  `-- Agent Gateway
        |-- Status Report
        |-- Event Report
        `-- Task Dispatch

Each Kubernetes Cluster
  |
  |-- Argo CD
  |-- PaaS Agent
  `-- Application Workloads
```

### 2.1 控制面职责

PaaS 控制面负责：

- 用户、租户、项目、角色、权限。
- 应用、环境、版本、晋级、部署记录。
- GitLab 托管源码仓库创建和权限同步。
- Jenkins Job 创建、构建触发、日志采集和结果回调。
- GitLab 部署清单仓库变更。
- 接收 PaaS Agent 上报的集群、Argo CD、Workload 状态。
- 聚合并展示用户可理解的应用状态。

控制面不负责：

- 直接操作 Kubernetes API Server。
- 暴露 Jenkins、Argo CD、部署清单仓库给普通用户。
- 直接保存 kubeconfig。
- 在集群外 watch Pod 或 Deployment。

### 2.2 数据面职责

数据面由每个 Kubernetes 集群内的 Argo CD 和 PaaS Agent 组成。

Argo CD 负责：

- 监听 GitLab 部署清单仓库。
- 将 Git 中的期望状态同步到 Kubernetes。
- 维护 Application 的 sync status、health status、operation state。

PaaS Agent 负责：

- 读取 Argo CD Application 状态。
- 读取必要的 Kubernetes Workload、Pod、Event 状态。
- 将底层状态转换成 PaaS 标准状态模型。
- 主动上报 PaaS 控制面。
- 拉取并执行受控任务，例如 Argo CD refresh/sync。

## 3. 核心领域模型

### 3.1 租户、用户和权限

核心对象：

```text
User
Identity
Tenant
TenantMember
Project
Group
GroupMember
ServiceAccount
Role
Permission
RoleBinding
AccessToken
AuditLog
```

设计规则：

- User 只表示登录主体，不直接保存业务权限。
- Identity 表示 User 的登录身份来源，当前测试可用版本支持 `local` 和 `oidc`。
- 本地用户由平台自主创建和管理，密码只保存强哈希结果，不保存明文密码。
- OIDC 用户通过 `issuer` + `subject` 唯一映射到平台 User，登录成功后按配置同步显示名、邮箱和头像等基础资料。
- 同一个 User 后续可以绑定多个 Identity；当前测试可用版本至少支持一个 User 绑定一个本地身份或一个 OIDC 身份。
- OIDC Provider 配置由平台管理员维护，包含 issuer、client_id、client_secret 引用、scope、redirect_uri 和启用状态；client_secret 不允许返回前端。
- Tenant 是最高资源隔离单位。
- Project 是租户下的应用分组和权限边界。
- Role 是权限集合。
- Permission 使用 `resource:action` 格式。
- RoleBinding 表示某个主体在某个作用域下拥有某个角色。
- 主体可以是 User、Group 或 ServiceAccount。

作用域：

```text
platform
tenant
project
application
environment
```

内置角色：

```text
platform_admin
tenant_owner
tenant_admin
project_admin
developer
viewer
operator
prod_approver
security_auditor
```

权限点示例：

```text
application:create
application:read
application:update
application:delete

environment:create
environment:read
environment:update
environment:delete

build:create
build:read
build:cancel

deployment:create
deployment:read
deployment:approve
deployment:rollback

secret:create
secret:update
secret:read_metadata

project:update
tenant:update
audit:read
```

权限校验流程：

```text
1. 定位资源所属 tenant/project/application/environment。
2. 查询用户直接 RoleBinding。
3. 查询用户所属 Group 的 RoleBinding。
4. 保留覆盖目标资源作用域的 RoleBinding。
5. 合并角色权限。
6. 判断是否包含目标 action。
7. 再检查环境策略、审批策略和发布窗口。
```

生产环境发布必须额外检查：

- 用户是否有生产发布或审批权限。
- 是否需要多人审批。
- 是否禁止自审批。
- 是否在允许发布窗口内。

登录流程：

```text
本地用户登录
  -> 提交账号和密码
  -> 校验本地 Identity 和密码哈希
  -> 生成访问令牌和刷新令牌
  -> 返回当前 User、Tenant/Project 上下文和权限摘要

OIDC 登录
  -> 前端跳转到 PaaS OIDC 发起接口
  -> PaaS 生成 state/nonce 并跳转企业身份源
  -> 身份源回调 PaaS callback
  -> PaaS 校验 code、state、nonce 和 id_token
  -> 按 issuer + subject 查找或创建 Identity/User
  -> 生成访问令牌和刷新令牌
```

### 3.2 应用模型

核心对象：

```text
Application
ApplicationSource
SourceRepository
Environment
EnvironmentConfig
EnvironmentSecret
EnvironmentRoute
EnvironmentClusterBinding
EnvironmentState
```

关系：

```text
Tenant
  `-- Project
        |-- SourceRepository
        `-- Application
              |-- ApplicationSource -> SourceRepository
              `-- Environment
                    |-- EnvironmentConfig
                    |-- EnvironmentSecret
                    |-- EnvironmentRoute
                    |-- EnvironmentClusterBinding
                    `-- EnvironmentState
```

Application 表示平台内最小独立交付单元，而不是默认表示一个完整业务系统。它应该能够独立构建、发布、扩缩容、回滚和观察状态，例如 `user-service`、`payment-api`、`admin-web`、`order-worker`。

如果用户口中的“应用”由多个微服务、前端和后端共同组成，平台应建模为同一个 Project 下的多个 Application。后续可以增加 ApplicationGroup、System 或 Product 概念用于导航和组合发布，但不改变 Application 作为交付单元的语义。

Application 默认产物是一个主容器镜像。Sidecar、exporter 或与主容器生命周期强绑定的辅助容器可以作为同一个 Application 的附属容器；多个可以独立发布的微服务不应合并成一个 Application。

SourceRepository 表示平台托管源码仓库。一个 SourceRepository 可以对应一个 Application，也可以被多个 Application 复用，以支持 monorepo：

```text
Single repo:
  SourceRepository user-service
    `-- Application user-service

Monorepo:
  SourceRepository commerce-platform
    |-- Application user-api    source_path=services/user-api
    |-- Application order-api   source_path=services/order-api
    `-- Application admin-web   source_path=frontend/admin-web
```

ApplicationSource 保存某个 Application 的一个代码源构建入口。一个 Application 可以有多个 ApplicationSource；第一个或 `is_primary=true` 的代码源作为默认展示和兼容主产物来源：

```text
app_id
source_key
source_repo_id
source_path
build_command
artifact_copy_command
runtime_base_image
artifact_deploy_path
default_ref
is_primary
```

当前测试可用版本通过构建环境支持 Java 构建，默认构建环境包括：

```text
java-springboot
java-tomcat
```

构建环境保存构建镜像。构建命令和产物拷贝命令由 ApplicationSource 的 BuildSpec 明确填写。镜像生成统一使用平台维护的运行时基础镜像，不要求用户提供 Dockerfile。

Environment 表示应用在某个环境中的运行上下文，例如：

```text
dev
test
staging
prod
```

Environment 应独立管理：

- 当前版本。
- 期望版本。
- 资源规格。
- 环境变量。
- 密钥引用。
- 域名和路由。
- 部署集群位置。
- 同步状态。
- 健康状态。
- 实例状态。

Environment 不等于 Kubernetes 集群。Environment 是逻辑运行环境，Cluster 是实际资源池，二者通过 EnvironmentClusterBinding 关联。一个 prod 环境可以绑定多个集群；如果平台尚未注册可用集群，环境仍然可以先创建，并处于 `pending_cluster_binding` 状态。

环境状态建议：

```text
draft
pending_cluster_binding
cluster_bound
deployable
deploying
running
degraded
disabled
```

状态含义：

- `draft`：环境刚创建，配置尚不完整。
- `pending_cluster_binding`：缺少集群绑定，不能部署。
- `cluster_bound`：已绑定集群和 namespace，但部署资源未完全准备。
- `deployable`：Argo CD Application 和清单路径已准备好，可以部署。
- `deploying`：正在部署。
- `running`：当前版本运行健康。
- `degraded`：运行异常。
- `disabled`：环境被手动禁用。

### 3.3 版本、变更包和晋级

采用类 Kargo 的环境晋级模型，不引入 Spinnaker 概念。

核心对象：

```text
Release
Freight
FreightItem
DeliveryFlow
DeliveryStage
Promotion
PromotionApproval
Deployment
DeploymentEvent
```

定义：

- Release：某个 Application 的一次构建结果，包含 commit、构建信息和主镜像。
- Freight：可交付变更包。当前测试可用版本中通常包含一个 Application 的一个 Release，后续可通过多个 FreightItem 组合多个 Application 的 Release、配置版本或迁移任务。
- FreightItem：Freight 内的具体项，例如 image、config、migration、application_release。
- DeliveryFlow：应用的交付流，例如 `dev -> test -> staging -> prod`。
- DeliveryStage：交付流中的一个阶段，通常绑定一个 Environment。
- Promotion：把某个 Freight 推进到某个 Stage。
- Deployment：Promotion 对应的实际部署和同步结果。

推荐状态：

```text
Promotion:
  created
  pending_approval
  approved
  rejected
  manifest_updating
  manifest_updated
  syncing
  healthy
  failed
  aborted

Deployment:
  pending
  syncing
  progressing
  succeeded
  failed
  degraded
  unknown
```

典型流程：

```text
构建成功
  -> 生成 Release
  -> 生成 Freight
  -> 自动发布到 dev
  -> 晋级到 test
  -> 晋级到 staging
  -> 申请晋级到 prod
  -> 审批通过
  -> 修改部署清单仓库
  -> Argo CD 同步
  -> Agent 上报 Healthy
```

核心对象：

```text
Cluster
ClusterHeartbeat
ClusterResourceSnapshot
EnvironmentClusterBinding
ClusterDeployment
ClusterTask
ClusterTaskResult
```

Cluster 字段建议：

```text
id
name
provider
region
zone
env
status
version
labels
last_heartbeat_at
```

Cluster 状态：

```text
ready
degraded
unreachable
draining
disabled
```

EnvironmentClusterBinding 表示某个环境与某个 Kubernetes 集群的绑定关系：

```text
env_id
cluster_id
namespace
replicas
traffic_weight
argo_project
argo_application_name
manifest_repo_id
manifest_path
status
```

Binding 状态：

```text
pending
active
syncing
failed
disabled
```

示例：

```text
Environment: prod
  |-- EnvironmentClusterBinding -> prod-shanghai-1
  `-- EnvironmentClusterBinding -> prod-beijing-1
```

如果创建应用时没有可用集群：

```text
Environment: dev      status = pending_cluster_binding
Environment: test     status = pending_cluster_binding
Environment: staging  status = pending_cluster_binding
Environment: prod     status = pending_cluster_binding
```

此时不创建 EnvironmentClusterBinding，也不创建 Argo CD Application；应用可以绑定已有源码仓库并在点击构建时创建构建流水线，但不能部署到这些环境。

当前测试可用版本策略：

- 创建应用时默认创建 dev/test/staging/prod 环境。
- 如果已有可用集群，可按模板自动创建 EnvironmentClusterBinding。
- 如果没有可用集群，环境进入 `pending_cluster_binding`，待集群注册后再绑定。
- 支持一个环境部署到一个或多个集群。
- 支持查看每个集群上的部署状态。
- 支持将集群标记为 draining 或 disabled。
- 不做自动跨集群故障迁移。

## 4. GitLab 集成

### 4.1 源码仓库策略

当前测试可用版本不允许用户直接导入外部源码仓库作为运行源。所有应用必须绑定平台托管的 SourceRepository。

创建 Application 时不创建源码仓库。SourceRepository 是 Project 下的独立资源，应先通过源码仓库管理或仓库迁移流程准备好。Application 创建流程只负责选择已有 SourceRepository，并创建 ApplicationSource 绑定。

源码仓库形态与交付单元形态解耦：

```text
一个 SourceRepository 可以服务一个 Application，也可以服务多个 Application。
一个 Application 是一个独立交付单元，当前测试可用版本默认生成一个主镜像。
```

用户创建应用时：

```text
1. PaaS 创建 Application。
2. PaaS 校验用户指定的 SourceRepository 属于当前 Project 或用户有权访问的 Project。
3. PaaS 创建 ApplicationSource，记录 source_path 和 BuildSpec。
4. PaaS 校验 source_path、build_command、artifact_copy_command、runtime_base_image 和 artifact_deploy_path。
5. PaaS 创建默认 Environment；Jenkins Job 延迟到用户点击构建时创建。
```

GitLab 组织结构：

```text
paas-managed
  `-- tenant-a
        `-- payment
              |-- user-service
              `-- order-service
```

映射：

```text
PaaS Tenant      -> GitLab Group
PaaS Project     -> GitLab Subgroup
PaaS SourceRepository -> GitLab Project
```

单仓单应用可以使用“一个 Application 对应一个 SourceRepository 和一个 GitLab Project”的结构，但 SourceRepository 需要在创建 Application 前已存在：

```text
paas-managed
  `-- tenant-a
        `-- payment
              `-- user-service
```

monorepo 场景下，多个 Application 复用同一个 SourceRepository：

```text
paas-managed
  `-- tenant-a
        `-- commerce
              `-- commerce-platform
                    |-- services/user-api
                    |-- services/order-api
                    `-- frontend/admin-web
```

SourceRepository 创建流程是独立流程：

```text
1. 用户在 Project 下创建 SourceRepository。
2. PaaS 在 GitLab 创建 Project。
3. PaaS 初始化空仓库或模板仓库。
4. PaaS 设置默认分支。
5. PaaS 设置分支保护。
6. PaaS 配置 Webhook。
7. PaaS 同步项目成员到 GitLab。
8. SourceRepository 标记为可用于创建 Application。
```

权限同步：

```text
tenant_owner  -> GitLab Owner
project_admin -> GitLab Maintainer
developer     -> GitLab Developer
viewer        -> GitLab Reporter
```

注意：

- GitLab 权限只代表源码权限。
- 构建、部署、生产审批仍由 PaaS 权限控制。
- 不把生产发布权限同步到 GitLab。

### 4.2 现有源码库迁移

用户可以把现有源码库迁移到 PaaS 托管仓库。

迁移不是“继续使用旧仓库”，而是复制代码到平台托管仓库，迁移完成后托管仓库成为标准源码。

支持模式：

```text
mirror: 保留完整 commit、branch、tag 历史
snapshot: 只迁移指定分支最新代码
```

当前测试可用版本优先实现 mirror：

```text
git clone --mirror <old_repo>
git push --mirror <managed_repo>
```

迁移流程：

```text
1. 用户填写源仓库地址和认证信息。
2. PaaS 创建托管 GitLab Project。
3. PaaS 创建 RepositoryMigration 任务。
4. Worker 拉取源仓库。
5. Worker 推送到托管仓库。
6. 校验默认分支、分支数量、tag 数量。
7. 分析项目语言和构建方式。
8. 生成可创建 Application 的目录和 BuildSpec 建议。
9. SourceRepository 标记为可用。
10. 用户后续在创建 Application 时指定该 SourceRepository，并确认 source_path、build_command、artifact_copy_command、runtime_base_image 和 artifact_deploy_path。
```

迁移任务状态：

```text
pending
creating_target_repo
cloning_source
pushing_target
verifying
analyzing
ready_for_application_binding
succeeded
failed
```

约束：

- 不自动迁移 Issue、MR、Wiki、CI 历史。
- 不自动照搬旧仓库成员。
- 旧仓库成员只能作为权限建议，由管理员确认后加入 PaaS Project。
- Git LFS 支持作为增强项，当前测试可用版本可先提示不支持或单独开关。

## 5. Jenkins 集成

### 5.1 构建环境

Jenkins 不暴露给用户。PaaS 通过 Jenkins API 创建和触发 Job。

当前测试可用版本由 PaaS 在创建或编辑 Application 时生成并保存 BuildSpec，不要求 Jenkins 自己适配所有源码结构。Jenkins 只运行平台统一模板，根据 BuildSpec 执行受限构建命令、收集产物，并用平台标准运行时镜像打包容器镜像。

当前测试可用版本支持 Java 应用构建，管理员通过构建环境维护构建镜像：

```text
java-springboot
java-tomcat
```

推荐使用平台托管 Job 模式：

```text
paas/{tenant_name}/{project_name}/{app_name}
```

点击构建时，PaaS 使用固定 Jenkins Job 名称。Job 不存在时创建，已存在时更新 Job 配置，然后触发构建；不再根据流水线配置生成 md5，也不再通过 md5 后缀创建新 Job。

Job 从统一模板生成，模板负责：

```text
1. 清理工作区。
2. 按每个 ApplicationSource 渲染独立检出阶段。
3. 按每个 ApplicationSource 渲染独立构建阶段，在 source_path 下执行 build_command。
4. 按每个 ApplicationSource 渲染独立产物收集阶段。
5. 执行平台受控打包入口生成镜像。
6. 回调 PaaS。
```

Job 参数：

```text
当前测试可用版本不向 Jenkins 传递构建参数。PaaS 在每次触发构建前按本次 BuildRun 渲染 Jenkinsfile，并更新固定 Jenkins Job。
```

BuildSpec 字段：

```text
source_path
build_command
artifact_copy_command
runtime_base_image
artifact_deploy_path
default_ref
```

BuildSpec 规则：

- `build_command` 由用户填写，但只能在 `source_path` 下执行。
- `artifact_copy_command` 是必填产物拷贝命令，必须把当前代码源构建产物复制到 `$PAAS_ARTIFACT_OUTPUT`。
- `runtime_base_image` 必须来自平台允许列表。
- `artifact_deploy_path` 是可选产物放置路径；填写时必须是绝对路径，且不允许包含 `..`。
- Web Console 创建应用时将 `runtime_base_image` 和 `artifact_deploy_path` 作为应用级运行时环境配置展示一次，不随 ApplicationSource 数量重复。
- 当前测试可用版本暂不支持客户仓库自带 Jenkinsfile。
- 当前测试可用版本不要求客户在源码仓库提供 Dockerfile；Dockerfile 由平台专用 Git 仓库统一维护，并由平台统一默认策略选择。

SourceRepository 迁移或扫描后只生成 BuildSpec 建议，不自动决定最终配置。扫描内容包括 `pom.xml`、`build.gradle`、`target/*.jar` 和 `target/*.war`。用户在创建 Application 时确认或调整建议后，PaaS 才写入 ApplicationSource。

BuildPipeline 模型：

```text
id
app_id
provider = jenkins
external_job_name
template_id
status
managed_by_platform
created_at
updated_at
```

### 6.1.1 应用部署模板定制

平台应提供默认应用部署模板，但不能让所有用户直接修改同一个共享模板。共享模板由平台维护，只作为初始化来源。

用户需要自定义 initContainer、目录创建、目录权限、volumeMount、安全上下文等场景时，PaaS 应为租户、项目或 Application 创建独立的部署模板副本或 overlay。

推荐模型：

```text
DeploymentTemplate
DeploymentTemplateRevision
```

`DeploymentTemplate` 表示一个可被用户维护的应用部署模板，归属范围可以是 tenant、project 或 application。当前测试可用版本优先支持 application 级模板。

`DeploymentTemplateRevision` 表示模板的一次版本化变更，记录模板内容、变更人、校验结果、commit/MR 信息和审计信息。

模板定制规则：

- 平台基础模板只读，不允许业务用户直接修改。
- 用户修改的是自己作用域内的模板副本或 overlay。
- 用户不直接访问 GitLab 部署清单仓库，所有修改通过 PaaS 完成。
- 模板变更必须经过语法校验、策略校验和审计记录。
- 模板发布到环境时，PaaS 通过 ManifestRevision 记录最终写入清单仓库的结果。
- 模板可以支持 Helm chart values、模板文件或 patch overlay；当前测试可用版本优先支持 application 级 Helm chart 副本或 overlay。
- 不同用户、租户、项目、Application 的模板互不影响。

示例定制场景：

```yaml
initContainers:
  - name: init-permission
    image: busybox:1.36
    command:
      - sh
      - -c
      - mkdir -p /data/logs && chmod 775 /data/logs
```

发布某个 Freight 到环境时，PaaS 修改对应环境的 values：

```yaml
image:
  repository: registry.example.com/user-service
  tag: v42
  digest: sha256:xxxx
```

## 7. PaaS Agent

### 7.1 Agent 职责

每个 Kubernetes 集群部署一个 PaaS Agent。

Agent 负责：

- 心跳上报。
- 集群资源快照上报。
- watch Argo CD Application。
- watch Deployment、StatefulSet、ReplicaSet、Pod、Event。
- 汇总应用状态。
- 上报部署事件。
- 拉取并执行控制面任务。

Agent 不负责：

- 执行业务发布决策。
- 任意修改 Kubernetes 业务资源。
- 保存平台用户凭据。

### 7.2 Agent API

Agent 到控制面：


### 7.3 实时性策略

采用事件推送加定时校准：

- watch 到 Argo CD 或 Workload 状态变化后立即上报。
- 每 30 秒上报一次状态快照。
- 每 10 秒上报一次心跳。
- Agent 离线超过阈值后，Cluster 标记为 `unreachable`。

### 7.4 Agent RBAC

最小权限：

- get/list/watch `argoproj.io/Application`。
- get/list/watch Deployment、StatefulSet、DaemonSet、ReplicaSet、Pod、Event。
- patch 指定 Argo CD Application annotation，用于 refresh。
- 不授予任意 apply/delete 业务资源权限。

## 8. 关键业务流程

### 8.1 创建应用

```text
1. 用户进入创建应用向导。
2. 填写应用名称、项目、类型、技术模板。
3. 确认该 Application 是一个独立交付组件，而不是多个微服务的集合。
4. 指定已有平台托管 SourceRepository。
5. 配置 source_path 和 BuildSpec，包括 build_command、artifact_copy_command、runtime_base_image 和 artifact_deploy_path。
6. 平台按环境模板默认选择 dev/test/staging/prod。
7. 如果有可用集群，用户可确认或调整环境到集群的绑定。
8. 如果没有可用集群，环境仍会创建，但状态为 pending_cluster_binding。
9. 提交创建。
10. PaaS 创建 Application 和 Environment。
11. PaaS 校验 SourceRepository 访问权限和路径配置。
12. PaaS 创建 ApplicationSource。
13. 对已绑定集群的环境，PaaS 创建 EnvironmentClusterBinding。
14. 对已绑定集群的环境，PaaS 创建部署清单目录和 Argo CD Application 清单。
15. 对未绑定集群的环境，PaaS 暂不创建 Argo CD Application。
16. PaaS 触发首次构建，可选；此时才创建或复用 Jenkins Job。
```

创建过程以异步任务展示：

```text
创建应用记录
校验源码仓库
创建 ApplicationSource
校验源码路径和 BuildSpec
创建构建流水线
创建默认环境
绑定集群
创建部署清单
注册 Argo CD 应用
触发首次构建
```

每一步必须可重试，并记录失败原因。

没有集群时的创建结果：

```text
Application: created
ApplicationSource: created
Environment: created, pending_cluster_binding
GitLab Source Repository: existing and associated
Jenkins BuildPipeline: created
Argo CD Application: not created
Deployment: blocked until cluster binding is active
```

集群注册后的补齐流程：

```text
1. 管理员注册 Cluster。
2. 平台扫描 pending_cluster_binding 环境。
3. 根据 ClusterBindingPolicy 或用户手动选择创建 EnvironmentClusterBinding。
4. 创建 namespace、manifest path 和 Argo CD Application。
5. 环境状态从 pending_cluster_binding 变为 deployable。
```

### 8.2 构建应用

```text
1. 用户点击构建。
2. PaaS 校验 build:create。
3. PaaS 查询 ApplicationSource。
4. PaaS 根据构建模板、ApplicationSource、BuildSpec、构建环境、运行时环境和平台 Dockerfile 仓库配置渲染 Jenkins Job；Job 不存在时创建，已存在时更新。
5. PaaS 创建 BuildRun。
6. PaaS 调用 Jenkins buildWithParameters。
7. PaaS 记录 queueId。
8. PaaS 轮询 queue item，拿到 build number。
9. PaaS 增量采集构建日志。
10. Jenkins 构建完成后回调 PaaS。
11. PaaS 更新 BuildRun。
12. PaaS 写入 BuildArtifact，当前测试可用版本标记一个主镜像。
13. PaaS 生成 Release 和 Freight。
```

### 8.3 晋级发布

```text
1. 用户选择 Freight。
2. 用户选择目标 Environment 或 DeliveryStage。
3. PaaS 校验 deployment:create。
4. 检查目标 Environment 是否存在 active EnvironmentClusterBinding。
5. 如果环境处于 pending_cluster_binding，阻止部署并提示先绑定集群。
6. 若目标为 prod，进入审批。
7. 审批通过后，PaaS 创建 Promotion。
8. PaaS 修改部署清单仓库。
9. PaaS 记录 ManifestRevision。
10. Argo CD 检测 Git 变化并同步。
11. Agent 上报 Syncing/Progressing。
12. Pod Ready 后，Agent 上报 Synced + Healthy。
13. PaaS 将 Deployment 标记为 succeeded。
```

### 8.4 回滚

回滚不直接操作 Argo CD 或 Kubernetes。

```text
1. 用户选择历史 Freight。
2. PaaS 创建 rollback Promotion。
3. PaaS 修改清单仓库到历史镜像 digest。
4. Argo CD 同步。
5. Agent 上报状态。
6. PaaS 记录回滚审计。
```

## 12. 安全与审计

安全要求：

- 所有用户请求经过 RBAC。
- GitLab、Jenkins、Argo CD 凭据不暴露给用户。
- 平台 Token 加密或哈希存储。
- 用户 Token 不长期用于后台自动化。
- 自动化操作使用平台 Bot 或服务账号。
- Secret 不允许明文展示，只允许创建、更新和引用。
- PaaS 控制面不保存 kubeconfig。
- Agent token 只允许上报所属 clusterId。
- Agent token 支持轮换。

审计事件：

```text
用户登录
创建租户
```

## 13. 测试可用版本范围

当前测试可用版本必须具备：

- 用户、租户、项目、RBAC。
- PaaS 托管源码仓库独立管理。
- 创建 Application 时必须指定已有 SourceRepository。
- Application 作为最小独立交付单元。
- 单仓库单应用，以及 monorepo 下通过 source_path 创建多个 Application。
- 现有仓库迁移到托管仓库。
- 点击构建时按固定名称创建或更新 Jenkins Job 并触发。
- 实时构建日志。
- BuildArtifact 主镜像记录。
- Release/Freight 生成，当前测试可用版本中一个 Freight 通常包含一个 Application Release。
- dev/test/staging/prod 环境。
- Promotion 晋级。
- prod 审批。
- GitLab 清单仓库修改。
- Argo CD GitOps 部署。
- PaaS Agent 状态上报。
- 应用列表、应用详情、环境详情、构建详情、发布晋级页面。

当前测试可用版本暂不纳入：

- 外部源码仓库直接作为运行源。
- 一个 Application 表示多个可独立发布微服务。
- 多 Application 的组合发布编排。
- 用户访问 Jenkins UI。
- 用户访问部署清单仓库。
- 用户访问 Argo CD UI。
- PaaS 直连 Kubernetes API Server。
- 自动跨集群故障迁移。
- 服务网格和复杂流量治理。
- 计费系统。
- Issue/MR/Wiki/CI 历史迁移。

### 14.1 单元测试

### 14.2 集成测试

### 14.3 端到端测试

覆盖用户路径：

```text
准备或迁移 SourceRepository
创建应用
首次构建

发布到 dev

审批 prod

```

### 14.4 故障测试

覆盖：

- 独立创建 SourceRepository 时 GitLab 创建仓库失败。
- 创建应用时 SourceRepository 不存在或无权限。
- 创建应用时 source_path、build_command、artifact_copy_command、runtime_base_image 或 artifact_deploy_path 不合法。
- 仓库迁移失败。
- 点击构建时 Jenkins Job 创建失败。
- Jenkins 构建失败。
- artifact_copy_command 没有向 `$PAAS_ARTIFACT_OUTPUT` 写入任何文件。
- 镜像推送失败。
- 构建日志流中断。
- 部署清单提交失败。
- Argo CD OutOfSync。
- Argo CD Degraded。
- Agent 离线。
- Pod ImagePullBackOff。
- 生产审批被拒绝。

## 15. 验收标准

测试可用版本验收时，必须满足：

- 用户能在 PaaS 独立创建或迁移平台托管 SourceRepository。
- 用户能基于已有 SourceRepository 创建 Application，并指定 source_path 和 BuildSpec。
- 用户能在 PaaS 触发 Jenkins 构建并实时查看日志。
- 构建成功后能生成可发布版本。
- 用户能将版本晋级到 dev/test/staging/prod。
- prod 发布必须经过审批。
- PaaS 通过修改部署清单触发 Argo CD 部署。
- PaaS 不直连 Kubernetes API Server。
- 集群内 Agent 能上报 Argo CD 和 Workload 状态。
- 用户能在 PaaS 看到应用环境状态、实例状态、发布结果和失败原因。
- 所有关键操作都有审计日志。

## 16. 默认假设

- PaaS、GitLab、Jenkins、Argo CD 均部署在企业内部网络。
- PaaS 主数据库使用 MySQL 8.0+，存储引擎为 InnoDB，字符集为 utf8mb4。
- GitLab 源码仓库和部署清单仓库可以使用同一 GitLab 实例的不同 Group。
- 每个 Kubernetes 集群独立部署 Argo CD。
- PaaS Agent 可以访问所在集群内 Kubernetes API Server。
- Jenkins 可以访问 GitLab 托管源码仓库和镜像仓库。
- 镜像仓库已经存在。
- 平台已经准备 Spring Boot 和 Tomcat 运行时基础镜像。
- 企业用户可以通过 OIDC 统一身份源登录 PaaS，也可以由平台管理员创建本地用户后使用账号密码登录。
- 生产发布权限只在 PaaS 内控制。
