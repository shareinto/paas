# PaaS 平台实现文档

## 1. 建设目标

本平台定位为企业内部应用交付 PaaS。用户只登录 PaaS 平台，即可完成应用创建、源码托管、构建、环境配置、版本晋级、发布审批、运行状态查看、日志查看和回滚。

平台底层集成 GitLab、Jenkins、Argo CD 和 Kubernetes，但这些系统不直接暴露给普通用户：

- GitLab：承载源码仓库和部署清单仓库。
- Jenkins：执行构建流水线。
- Argo CD：基于 GitOps 将部署清单同步到 Kubernetes。
- Kubernetes：应用运行面。
- PaaS Agent：部署在每个 Kubernetes 集群中，负责状态采集和受控任务执行。

核心原则：

- PaaS 是统一入口、权限中心和业务编排中心。
- GitLab 是源码和部署期望状态的来源。
- Jenkins 只负责构建，不作为用户入口。
- Argo CD 只负责部署同步，不作为用户入口。
- PaaS 控制面不直连 Kubernetes API Server。
- 每个集群内的 PaaS Agent 主动连接 PaaS 控制面，上报状态。

## 2. 总体架构

```text
User Browser
  |
  v
PaaS Web Console
  |
  v
PaaS Control Plane
  |
  |-- PaaS Database (MySQL 8.0+)
  |-- GitLab Adapter
  |     |-- Source Repository API
  |     `-- Manifest Repository API
  |-- Jenkins Adapter
  |     |-- Job Management
  |     |-- Build Trigger
  |     `-- Progressive Build Log
  |-- Argo/GitOps Adapter
  |     `-- Manifest Change / MR / Commit
  `-- Agent Gateway
        |-- Status Report
        |-- Event Report
        `-- Task Dispatch

Each Kubernetes Cluster
  |
  |-- Argo CD
  |-- PaaS Agent
  `-- Application Workloads
```

### 2.1 控制面职责

PaaS 控制面负责：

- 用户、租户、项目、角色、权限。
- 应用、环境、版本、晋级、部署记录。
- GitLab 已有源码仓库登记、扫描和权限同步。
- Jenkins Job 创建、构建触发、日志采集和结果回调。
- GitLab 部署清单仓库变更。
- 接收 PaaS Agent 上报的集群、Argo CD、Workload 状态。
- 聚合并展示用户可理解的应用状态。

控制面不负责：

- 直接操作 Kubernetes API Server。
- 暴露 Jenkins、Argo CD、部署清单仓库给普通用户。
- 直接保存 kubeconfig。
- 在集群外 watch Pod 或 Deployment。

### 2.2 数据面职责

数据面由每个 Kubernetes 集群内的 Argo CD 和 PaaS Agent 组成。

Argo CD 负责：

- 监听 GitLab 部署清单仓库。
- 将 Git 中的期望状态同步到 Kubernetes。
- 维护 Application 的 sync status、health status、operation state。

PaaS Agent 负责：

- 读取 Argo CD Application 状态。
- 读取必要的 Kubernetes Workload、Pod、Event 状态。
- 将底层状态转换成 PaaS 标准状态模型。
- 主动上报 PaaS 控制面。
- 拉取并执行受控任务，例如 Argo CD refresh/sync。

## 3. 核心领域模型

### 3.1 租户、用户和权限

核心对象：

```text
User
Identity
Tenant
TenantMember
Project
Group
GroupMember
ServiceAccount
Role
Permission
RoleBinding
AccessToken
AuditLog
```

设计规则：

- User 只表示登录主体，不直接保存业务权限。
- Identity 表示 User 的登录身份来源，当前测试可用版本支持 `local` 和 `oidc`。
- 本地用户由平台自主创建和管理，密码只保存强哈希结果，不保存明文密码。
- OIDC 用户通过 `issuer` + `subject` 唯一映射到平台 User，登录成功后按配置同步显示名、邮箱和头像等基础资料。
- 同一个 User 后续可以绑定多个 Identity；当前测试可用版本至少支持一个 User 绑定一个本地身份或一个 OIDC 身份。
- OIDC Provider 配置由平台管理员维护，包含 issuer、client_id、client_secret 引用、scope、redirect_uri 和启用状态；client_secret 不允许返回前端。
- Tenant 是最高资源隔离单位。
- Project 是租户下的应用分组和权限边界。
- Role 是权限集合。
- Permission 使用 `resource:action` 格式。
- RoleBinding 表示某个主体在某个作用域下拥有某个角色。
- 主体可以是 User、Group 或 ServiceAccount。

作用域：

```text
platform
tenant
project
application
environment
```

内置角色：

```text
platform_admin
tenant_owner
tenant_admin
project_admin
developer
viewer
operator
prod_approver
security_auditor
```

权限点示例：

```text
application:create
application:read
application:update
application:delete

environment:create
environment:read
environment:update
environment:delete

build:create
build:read
build:cancel

deployment:create
deployment:read
deployment:approve
deployment:rollback

secret:create
secret:update
secret:read_metadata

project:update
tenant:update
audit:read
```

权限校验流程：

```text
1. 定位资源所属 tenant/project/application/environment。
2. 查询用户直接 RoleBinding。
3. 查询用户所属 Group 的 RoleBinding。
4. 保留覆盖目标资源作用域的 RoleBinding。
5. 合并角色权限。
6. 判断是否包含目标 action。
7. 再检查Stage 策略、审批策略和发布窗口。
```

生产 Stage 发布必须额外检查：

- 用户是否有生产发布或审批权限。
- 是否需要多人审批。
- 是否禁止自审批。
- 是否在允许发布窗口内。

登录流程：

```text
本地用户登录
  -> 提交账号和密码
  -> 校验本地 Identity 和密码哈希
  -> 生成访问令牌和刷新令牌
  -> 返回当前 User、Tenant/Project 上下文和权限摘要

OIDC 登录
  -> 前端跳转到 PaaS OIDC 发起接口
  -> PaaS 生成 state/nonce 并跳转企业身份源
  -> 身份源回调 PaaS callback
  -> PaaS 校验 code、state、nonce 和 id_token
  -> 按 issuer + subject 查找或创建 Identity/User
  -> 生成访问令牌和刷新令牌
```

### 3.2 应用模型

核心对象：

```text
Application
ApplicationSource
SourceRepository
Workload
WorkloadStageConfig
ApplicationStageState
DeliveryFlowTemplateStage
StageClusterBinding
```

关系：

```text
Tenant
  `-- Project
        |-- SourceRepository
        `-- Application
              |-- ApplicationSource -> SourceRepository
              |-- Workload
              |     `-- WorkloadStageConfig
              `-- AppStage projection -> DeliveryFlowTemplateStage
```

Application 表示平台内的业务交付上下文，可以对应一个业务应用、服务组合或产品功能边界。Application 不再等同于单个容器镜像或单个 Kubernetes 工作负载。

Workload 表示最小可部署单元。一个 Workload 对应一个容器镜像和一个 Kubernetes 工作负载，当前支持 Deployment 和 StatefulSet。例如一个 `payment` Application 可以包含 `admin-web`、`payment-api` 和 `settlement-worker` 三个 Workload。

Application 下 Workload 数量由用户动态增删。创建 Freight 时至少需要一个启用 Workload，并且必须覆盖该 Application 下全部启用 Workload。

SourceRepository 表示平台已登记的源码仓库。一个 SourceRepository 可以对应一个 Application，也可以被多个 Application 复用，以支持 monorepo：

```text
Single repo:
  SourceRepository user-service
    `-- Application user-service
          `-- Workload user-service

Monorepo:
  SourceRepository commerce-platform
    `-- Application commerce
          |-- Workload user-api    source_path=services/user-api
          |-- Workload order-api   source_path=services/order-api
          `-- Workload admin-web   source_path=frontend/admin-web
```

ApplicationSource 保存某个 Workload 的一个代码源构建入口。一个 Workload 可以有多个 ApplicationSource；第一个或 `is_primary=true` 的代码源作为默认展示和兼容主产物来源：

```text
app_id
workload_id
source_key
source_repo_id
source_path
build_command
artifact_copy_command
runtime_base_image
artifact_deploy_path
default_ref
is_primary
```

当前测试可用版本通过构建环境支持 Java/Node 相关构建预设，初始化默认构建环境包括：

```text
gradle7-jdk11
node22
```

初始化默认运行环境包括：

```text
springboot-jdk11
tomcat-jdk11
nginx1221
```

构建环境保存构建镜像。构建命令和产物拷贝命令由 ApplicationSource 的 BuildSpec 明确填写。镜像生成统一使用平台维护的运行时基础镜像和 Dockerfile 路径，不要求用户提供 Dockerfile。

Stage 表示租户交付流中的运行阶段，例如：

```text
dev
test
staging
prod
```

Stage 不作为应用私有数据复制保存。应用详情页中的 Stage 是 DeliveryFlowTemplateStage 的实时投影，运行状态按 `application_id + stage_key` 保存：

- 当前版本。
- 期望版本。
- 部署集群位置。
- 同步状态。
- 健康状态。
- 实例状态。

Workload 默认配置用于管理该 Workload 的 Helm values 默认值。创建或编辑工作负载时维护的是默认配置，不绑定具体环境或 Stage；发布时如果目标 Stage 没有覆盖配置，GitOps values 使用该默认配置。

WorkloadStageConfig 应独立管理每个 Workload 在每个 Stage 下的部署差异：

- 工作负载类型，Deployment 或 StatefulSet。
- 端口、服务暴露方式和域名路由。
- 资源规格、探针、实例数和滚动策略。
- 环境变量、密钥引用、配置文件和 values 覆盖。
- 可写目录、volume、volumeMount、initContainer 和安全上下文。

部署页的 Stage 卡片可以编辑某个 Workload 在该 Stage 下的覆盖配置。覆盖配置只对该 Stage 生效，不回写 Workload 默认配置。

Stage 不等于 Kubernetes 集群。Stage 是租户级交付阶段，Cluster 是实际资源池，二者通过 StageClusterBinding 关联。应用不再创建独立运行环境模型。

Stage 状态建议：

```text
draft
pending_cluster_binding
cluster_bound
deployable
deploying
running
degraded
disabled
```

状态含义：

- `draft`：环境刚创建，配置尚不完整。
- `pending_cluster_binding`：缺少集群绑定，不能部署。
- `cluster_bound`：已绑定集群和 namespace，但部署资源未完全准备。
- `deployable`：Argo CD Application 和清单路径已准备好，可以部署。
- `deploying`：正在部署。
- `running`：当前版本运行健康。
- `degraded`：运行异常。
- `disabled`：环境被手动禁用。

### 3.3 版本、变更包和晋级

采用类 Kargo 的环境晋级模型，不引入 Spinnaker 概念。

核心对象：

```text
Release
ImageBundle
Freight
FreightItem
DeliveryFlowTemplate
DeliveryFlowTemplateStage
DeliveryFlowTemplateEdge
StageClusterBinding
AppStage
DeliveryFlow
DeliveryStage
Promotion
PromotionApproval
FreightApproval
StageVerification
Deployment
DeploymentEvent
```

定义：

- Release：某个 Workload 的一次可发布镜像版本候选，包含 commit、构建信息和镜像，可引用 ImageBundle。
- ImageBundle：同一 Workload、同一 Build、同一编译产物对应的一组后台运行时镜像。每个镜像来自同一个逻辑运行时环境下的不同基础镜像，并携带仅后台可见的 selector labels。
- Freight：Application 级可交付变更包。一个 Freight 至少包含一个 FreightItem，且必须覆盖该 Application 下所有启用 Workload。
- FreightItem：Freight 内的 Workload 镜像版本项，每个条目绑定一个 Workload 和一个镜像版本。来源可以是 `pipeline_artifact` 或 `custom_image`；流水线产物来源可引用 ImageBundle。
- DeliveryFlowTemplate：租户级交付流 DAG 模板，定义 Stage 列表、依赖边、展示顺序、固定画布槽位、自动列色、审批角色、验证角色、是否需要部署前审批和部署后验证。
- DeliveryFlowTemplateStage：模板中的 Stage 定义，Stage key 创建后保持稳定；删除 Stage 会物理删除模板项和该 Stage 的集群绑定。
- DeliveryFlowTemplateEdge：模板中的有向依赖边，连接上游 Stage 和下游 Stage；保存时必须校验无环，允许多个根 Stage。
- StageClusterBinding：租户级 Stage 与 Kubernetes 集群的绑定关系；一个 Stage 最多绑定一个集群，一个集群可以绑定到多个 Stage。
- AppStage：Application 按租户模板生成的应用 Stage，是 Web Console 发布视角的主要对象。
- DeliveryFlow：应用的交付流；默认兼容 `dev -> test -> staging -> prod`，实际晋级条件以租户 DeliveryFlowTemplate DAG 为准。
- DeliveryStage：交付流中的一个阶段，当前可映射到同名 Stage。
- Promotion：把某个 Freight 推进到某个 Stage；Stage 化发布携带 `target_stage_key`，目标集群由 StageClusterBinding 唯一确定，兼容记录 `target_cluster_ids` 和 `namespace_override`。
- FreightApproval：Freight 级审批记录，部署前或晋级前使用。
- StageVerification：Stage 级人工验证记录，发布后使用；验证只要求已有部署记录，不强制依赖健康状态。
- Deployment：Promotion 对应的实际部署和同步结果。

推荐状态：

```text
Promotion:
  created
  pending_approval
  approved
  rejected
  manifest_updating
  manifest_updated
  syncing
  healthy
  failed
  aborted

Deployment:
  pending
  syncing
  progressing
  succeeded
  failed
  degraded
  unknown
```

典型流程：

```text
构建成功
  -> 生成 Workload Release 候选
  -> 自动尝试创建完整 Freight
  -> 本次构建 Workload 使用本次产物，其他 Workload 使用最近成功产物补齐
  -> 为当前 eligible Stage 自动创建手动发布 Promotion
  -> 用户确认发布
  -> 发布到 dev
  -> 晋级到 test
  -> 晋级到 staging
  -> 申请晋级到 prod
  -> 审批通过
  -> 修改部署清单仓库
  -> Argo CD 同步
  -> Agent 上报 Healthy
```

### 3.4 多集群模型

核心对象：

```text
Cluster
ClusterHeartbeat
ClusterResourceSnapshot
StageClusterBinding
ClusterDeployment
ClusterTask

```

Cluster 字段建议：

```text
id

provider

zone

status

labels

```

Cluster labels 用于部署期镜像选择，例如 `cloud=aliyun`、`cloud=aws`。标签只表达运行位置或运行时能力，不替代 StageClusterBinding。Cluster labels 和运行时镜像 selector labels 只在后台配置中可见，不在发布/部署用户流程中展示。

Cluster 状态：

```text
ready
degraded
unreachable
draining
disabled
```

StageClusterBinding 表示某个 Stage 与某个 Kubernetes 集群的绑定关系：

```text
stage_key
cluster_id
namespace
replicas
traffic_weight
argo_project
argo_application_name
manifest_repo_id
manifest_path
status
```

Binding 状态：

```text
pending
active
syncing
failed
disabled
```

示例：

```text
Stage: prod
  |-- StageClusterBinding -> prod-shanghai-1
  `-- StageClusterBinding -> prod-beijing-1
```

如果创建应用时没有可用集群：

```text
Stage: dev      status = pending_cluster_binding
Stage: test     status = pending_cluster_binding
Stage: staging  status = pending_cluster_binding
Stage: prod     status = pending_cluster_binding
```

此时不创建 StageClusterBinding，也不创建 Argo CD Application；应用可以先完成创建，用户后续可在应用详情中创建绑定源码仓库的构建流水线，但不能部署到这些 Stage。

当前测试可用版本策略：

- 创建应用时默认从 DeliveryFlow 模板投影 dev/test/staging/prod Stage。
- 如果已有可用集群，可按模板自动创建 StageClusterBinding。
- 如果没有可用集群，Stage 进入 `pending_cluster_binding`，待集群注册后再绑定。
- 支持一个 Stage 部署到一个或多个集群。
- 支持查看每个集群上的部署状态。
- 支持将集群标记为 draining 或 disabled。
- 不做自动跨集群故障迁移。

## 4. GitLab 集成

### 4.1 源码仓库策略

当前测试可用版本不要求构建流水线绑定已登记的 SourceRepository。源码仓库由用户先自行创建，PaaS 在 BuildPipeline 代码源配置中直接保存源码类型、源码地址和构建引用。

创建 Application 时不创建源码仓库，也不在创建应用流程中绑定源码仓库。Application 创建流程只负责创建应用记录；应用 Stage 来自租户 DeliveryFlow 模板实时投影，代码源、BuildSpec 和运行时环境在创建 BuildPipeline 时配置。

源码仓库形态与交付单元形态解耦：

```text
一个 SourceRepository 可以服务一个 Application，也可以服务多个 Application。
一个 Application 是一个独立交付单元，当前测试可用版本默认让用户选择一个逻辑运行时环境。若该逻辑运行时环境配置了多个后台镜像，构建成功后可生成一个 ImageBundle；ImageBundle 内多个镜像共享同一编译产物、commit 和验证结论。
```

用户创建应用和构建流水线时：

```text
1. PaaS 创建 Application。
2. 应用详情页实时投影租户 DeliveryFlow 模板 Stage。
3. 用户在应用详情中创建 BuildPipeline，填写源码类型、源码地址、构建引用、构建环境和运行时环境。
4. 用户在创建或编辑 Workload 时选择关联 BuildPipeline；一条 BuildPipeline 可以被多个 Workload 关联。
5. Git 源码可通过 GitLab 适配器按地址预览分支；SVN 源码使用完整 checkout 地址，revision 可选，默认 HEAD。
6. PaaS 创建 BuildPipelineSource，记录 source_path 和 BuildSpec，并按流水线逻辑运行时环境快照填充 runtime_base_image 和 artifact_deploy_path；dockerfile_path 与 selector labels 来自后台运行时镜像配置，不在用户流程展示。
7. Jenkins Job 延迟到用户点击构建时创建。
```

GitLab 组织结构：

```text
paas-managed
  `-- tenant-a
        `-- payment
              |-- user-service
              `-- order-service
```

映射：

```text
PaaS Tenant      -> GitLab Group
PaaS Project     -> GitLab Subgroup
PaaS SourceRepository -> GitLab Project
```

单仓单应用可以使用“一个 Application 的 BuildPipeline 对应一个 SourceRepository 和一个 GitLab Project”的结构，但 SourceRepository 作为项目资源独立存在：

```text
paas-managed
  `-- tenant-a
        `-- payment
              `-- user-service
```

monorepo 场景下，多个 Application 复用同一个 SourceRepository：

```text
paas-managed
  `-- tenant-a
        `-- commerce
              `-- commerce-platform
                    |-- services/user-api
                    |-- services/order-api
                    `-- frontend/admin-web
```

SourceRepository 创建流程是独立流程：

```text
1. 用户先在 GitLab 中自行创建源码仓库。
2. 用户在 Project 下登记 SourceRepository，填写仓库 HTTP 地址和默认分支。
3. PaaS 校验 HTTP 地址属于当前配置的 GitLab，解析 GitLab Project。
4. PaaS 扫描默认分支文件树；仓库不存在、无权限或默认分支不可读时拒绝登记。
5. PaaS 保存 SourceRepository 的 `http_url`、`ssh_url`、`git_project_id` 并标记为可用于创建 BuildPipeline。
```

权限同步：

```text
tenant_owner  -> GitLab Owner
project_admin -> GitLab Maintainer
developer     -> GitLab Developer
viewer        -> GitLab Reporter
```

注意：

- GitLab 权限只代表源码权限。
- 构建、部署、生产审批仍由 PaaS 权限控制。
- 不把生产发布权限同步到 GitLab。

### 4.2 现有源码库迁移

当前测试可用版本不再提供历史源码迁移流程。已有源码库如需接入平台，应由用户或管理员准备好 Git 或 SVN 源码地址，并在 BuildPipeline 代码源配置中直接填写。历史 RepositoryMigration API 保留兼容入口，但返回“不再支持源码迁移，请在流水线中直接填写源码地址”。

迁移任务状态：

```text
pending
creating_target_repo
cloning_source
pushing_target
verifying
analyzing
ready_for_application_binding
succeeded
failed
```

约束：

- 不自动迁移 Issue、MR、Wiki、CI 历史。
- 不自动照搬旧仓库成员。
- 旧仓库成员只能作为权限建议，由管理员确认后加入 PaaS Project。
- Git LFS 支持作为增强项，当前测试可用版本可先提示不支持或单独开关。

## 5. Jenkins 集成

### 5.1 构建环境

Jenkins 不暴露给用户。PaaS 通过 Jenkins API 创建和触发 Job。

当前测试可用版本由 PaaS 在创建或编辑 BuildPipeline 时生成并保存 BuildSpec，不要求 Jenkins 自己适配所有源码结构。Jenkins 只运行平台统一模板，根据 BuildSpec 执行受限构建命令、收集产物，并用平台标准运行时镜像打包容器镜像。

当前测试可用版本支持 Java/Node 相关构建预设，管理员通过构建环境维护构建镜像：

```text
gradle7-jdk11
node22
```

推荐使用平台托管 Job 模式：

```text
paas/{tenant_name}/{project_name}/{app_name}
```

点击构建时，PaaS 使用固定 Jenkins Job 名称。Job 不存在时创建，已存在时更新 Job 配置，然后触发构建；不再根据流水线配置生成 md5，也不再通过 md5 后缀创建新 Job。

Job 从统一模板生成，模板负责：

```text
1. 清理工作区。
2. 按每个 ApplicationSource 渲染独立检出阶段。
3. 按每个 ApplicationSource 渲染独立构建阶段，在 source_path 下执行 build_command。
4. 按每个 ApplicationSource 渲染独立产物收集阶段。
5. 执行平台受控打包入口生成镜像。
6. 回调 PaaS。
```

Job 参数：

```text
当前测试可用版本不向 Jenkins 传递构建参数。PaaS 在每次触发构建前按本次 BuildRun 渲染 Jenkinsfile，并更新固定 Jenkins Job。
```

BuildSpec 字段：

```text
source_path
build_command
artifact_copy_command
runtime_base_image
artifact_deploy_path
default_ref
```

BuildSpec 规则：

- `build_command` 由用户填写，但只能在 `source_path` 下执行。
- `artifact_copy_command` 是必填产物拷贝命令，必须把当前代码源构建产物复制到 `$PAAS_ARTIFACT_OUTPUT`。
- `runtime_base_image` 必须来自 BuildPipeline 选择的运行时环境。
- 逻辑运行时环境下可配置多个后台镜像，每个后台镜像可配置 selector labels；这些 labels 会进入系统内部快照、BuildArtifact 和 ImageBundle 镜像，用于后续按集群标签选择镜像，但不在创建流水线、构建、发布和部署用户流程中展示。
- `artifact_deploy_path` 是可选产物放置路径；填写时必须是绝对路径，且不允许包含 `..`。
- Web Console 创建流水线时将逻辑运行时环境作为流水线级配置展示一次，不随 BuildPipelineSource 数量重复；用户只能选择一个运行时环境，后台镜像目标由平台自动展开。
- 当前测试可用版本暂不支持客户仓库自带 Jenkinsfile。
- 当前测试可用版本不要求客户在源码仓库提供 Dockerfile；Dockerfile 由平台专用 Git 仓库统一维护，并由平台统一默认策略选择。

SourceRepository 迁移或扫描后只生成 BuildSpec 建议，不自动决定最终配置。扫描内容包括 `pom.xml`、`build.gradle`、`target/*.jar` 和 `target/*.war`。用户在创建 BuildPipeline 时确认或调整建议后，PaaS 才写入流水线代码源配置。

### 6.1.1 应用部署模板定制

平台应提供默认应用部署模板，但不能让所有用户直接修改同一个共享模板。共享模板由平台维护，只作为初始化来源。

用户需要自定义 initContainer、目录创建、目录权限、volumeMount、安全上下文等场景时，PaaS 应通过 Workload 部署配置和标准 Helm chart values 表达。平台基础 chart 只读，业务侧只维护 values 或受控 overlay。

推荐模型：

```text
DeploymentTemplate
DeploymentTemplateRevision
```

`DeploymentTemplate` 表示一个可被用户维护的部署模板，归属范围可以是 tenant、project、application 或 workload。当前测试可用版本优先支持 Workload 级 values 配置，平台基础 chart 由平台维护。

`DeploymentTemplateRevision` 表示模板的一次版本化变更，记录模板内容、变更人、校验结果、commit/MR 信息和审计信息。

模板定制规则：

- 平台基础模板只读，不允许业务用户直接修改。
- 用户修改的是自己作用域内的模板副本或 overlay。
- 用户不直接访问 GitLab 部署清单仓库，所有修改通过 PaaS 完成。
- 模板变更必须经过语法校验、策略校验和审计记录。
- 模板发布到环境时，PaaS 通过 ManifestRevision 记录最终写入清单仓库的结果。
- 模板可以支持 Helm chart values、模板文件或 patch overlay；当前版本优先使用单 Workload 标准 Helm chart 和 WorkloadStageConfig 生成 values。
- 不同用户、租户、项目、Application、Workload 的配置互不影响。

示例定制场景：

```yaml
initContainers:
  - name: init-permission
    image: busybox:1.36
    command:
      - sh
      - -c
      - mkdir -p /data/logs && chmod 775 /data/logs
```

发布某个 Freight 到 Stage时，PaaS 为 Freight 中每个 FreightItem 修改对应 Workload 的环境 values。Freight 只携带镜像版本，dev/prod 环境变量等差异保存在 WorkloadStageConfig 或环境 values 中：

如果 FreightItem 引用 ImageBundle，PaaS 在写入每个目标集群 values 前先按目标 Cluster labels 选择具体镜像。选择规则为：ImageBundle 镜像的 selector labels 必须全部匹配目标 Cluster labels；每个目标集群必须且只能匹配到一个镜像，否则发布失败并提示当前版本不适用于目标环境，请联系平台管理员检查运行时镜像配置。Helm chart、Argo CD 和 PaaS Agent 不承担 ImageBundle 选择逻辑，发布/部署用户流程不展示镜像标签和集群标签。

```yaml
image:
  repository: registry.example.com/user-service
  tag: v42
  digest: sha256:xxxx
```

## 7. PaaS Agent

### 7.1 Agent 职责

每个 Kubernetes 集群部署一个 PaaS Agent。

Agent 负责：

- 心跳上报。
- 集群资源快照上报。
- watch Argo CD Application。
- watch Deployment、StatefulSet、ReplicaSet、Pod、Event。
- 汇总应用状态。
- 上报部署事件。
- 拉取并执行控制面任务。

Agent 不负责：

- 执行业务发布决策。
- 任意修改 Kubernetes 业务资源。
- 保存平台用户凭据。

### 7.3 实时性策略

采用事件推送加定时校准：

- watch 到 Argo CD 或 Workload 状态变化后立即上报。
- 每 30 秒上报一次状态快照。
- 每 10 秒上报一次心跳。
- Agent 离线超过阈值后，Cluster 标记为 `unreachable`。

### 7.4 Agent RBAC

最小权限：

- get/list/watch `argoproj.io/Application`。
- get/list/watch Deployment、StatefulSet、DaemonSet、ReplicaSet、Pod、Event。
- patch 指定 Argo CD Application annotation，用于 refresh。
- 不授予任意 apply/delete 业务资源权限。

## 8. 关键业务流程

### 8.1 创建应用

```text
1. 用户进入创建应用向导。
2. 从项目工作台进入时自动带入所属项目；从全局入口进入时先选择项目。
3. 填写应用名称、显示名称、类型和描述。
4. 确认该 Application 是一个独立交付组件，而不是多个微服务的集合。
5. 平台按环境模板默认选择 dev/test/staging/prod。
6. 如果有可用集群，用户可确认或调整环境到集群的绑定。
7. 如果没有可用集群，Stage 投影仍可见，但状态为 pending_cluster_binding。
8. 提交创建。
9. PaaS 创建 Application，Stage 列表来自租户 DeliveryFlow 模板投影。
10. 用户在应用详情中创建 Workload。
11. 用户在应用详情中创建构建流水线，指定 Workload、源码类型、源码地址、构建引用、构建环境、运行时环境和 BuildSpec。
12. Git 源码可按地址刷新分支；SVN 源码可选 revision。PaaS 校验源码地址、源码路径和构建配置。
13. PaaS 创建 BuildPipelineSource 或兼容的 ApplicationSource 记录。
14. 对已绑定集群的环境，PaaS 创建 StageClusterBinding。
15. 对已绑定集群的环境，PaaS 创建部署清单目录和 Argo CD Application 清单。
16. 对未绑定集群的环境，PaaS 暂不创建 Argo CD Application。
17. PaaS 触发首次构建，可选；此时才创建或复用 Jenkins Job。
```

创建过程以异步任务展示：

```text
创建应用记录
投影默认 Stage
绑定集群
创建部署清单
注册 Argo CD 应用
```

构建流水线创建过程单独展示：

```text
校验 Workload
校验源码仓库
校验源码路径和 BuildSpec
创建构建流水线
创建代码源绑定
触发首次构建
```

每一步必须可重试，并记录失败原因。

没有集群时的创建结果：

```text
Application: created
Stage: created, pending_cluster_binding
GitLab Source Repository: project resource, not bound yet
Jenkins BuildPipeline: not created yet
Argo CD Application: not created
Deployment: blocked until cluster binding is active
```

集群注册后的补齐流程：

```text
1. 管理员注册 Cluster。
2. 平台扫描 pending_cluster_binding Stage。
3. 根据 ClusterBindingPolicy 或用户手动选择创建 StageClusterBinding。
4. 创建 namespace、manifest path 和 Argo CD Application。
5. Stage 状态从 pending_cluster_binding 变为 deployable。
```

### 8.2 构建应用

```text
1. 用户点击构建。
2. PaaS 校验 build:create。
3. PaaS 查询 ApplicationSource。
4. PaaS 根据构建模板、ApplicationSource、BuildSpec、构建环境、运行时环境和平台 Dockerfile 仓库配置渲染 Jenkins Job；Job 不存在时创建，已存在时更新。
5. PaaS 创建 BuildRun。
6. PaaS 调用 Jenkins buildWithParameters。
7. PaaS 记录 queueId。
8. PaaS 轮询 queue item，拿到 build number。
9. PaaS 增量采集构建日志。
10. Jenkins 构建完成后回调 PaaS。
11. PaaS 更新 BuildRun。
12. PaaS 写入 BuildArtifact，标记所属 Workload 和主镜像。
13. PaaS 为对应 Workload 生成 Release 候选版本。
```

### 8.3 晋级发布

```text
1. 用户进入发布晋级页。
2. Web Console 展示按租户模板投影生成的应用 Stage DAG。
3. 用户将 Freight 从时间轴拖拽到目标 Stage 卡片。
4. Web Console 校验该 Freight 是否可发布到目标 Stage，并在 Stage 卡片内展示内联确认和 Workload 镜像明细。
5. PaaS 校验 deployment:create。
6. PaaS 校验 Freight 是否覆盖所有启用 Workload。
7. PaaS 根据租户 DeliveryFlowTemplate DAG 校验 Freight 是否已经通过目标 Stage 的全部直接上游 Stage；如果直接上游要求部署后验证，则对应 StageVerification 必须通过。
8. PaaS 校验目标 Stage 存在唯一 active StageClusterBinding。
9. 如果目标 Stage 没有可用集群绑定，阻止部署并提示先在租户交付流模板中绑定集群。
10. 若目标 Stage 需要部署前审批，Freight 审批通过后才允许继续。
11. PaaS 创建 Promotion，并记录 `target_stage_key`、Stage 绑定集群和 `namespace_override`。
12. 发布或审核弹窗预览最终渲染后的 Kubernetes `manifests.yaml` 差异；不得向用户展示 `source_fingerprint`、内部 snapshot 或其他实现细节字段。
13. PaaS 按 FreightItem 更新各 Workload 的 Stage/集群 values；values 优先使用该 Stage 的 WorkloadStageConfig，未配置时回退到 Workload 默认配置；如果 FreightItem 引用 ImageBundle，先按目标集群 labels 解析出具体镜像。
14. PaaS 为目标集群记录 Deployment 和 ManifestRevision。
15. Argo CD 检测 Git 变化并同步。
16. Agent 上报 Syncing/Progressing。
17. Pod Ready 后，Agent 上报 Synced + Healthy。
18. 如 Stage 要求部署后验证，用户在 Stage 卡片打开人工验证弹窗并提交 StageVerification。
19. PaaS 将 Deployment 标记为 succeeded。
```

### 8.4 回滚

回滚不直接操作 Argo CD 或 Kubernetes。

```text
1. 用户选择历史 Freight。
2. PaaS 创建 rollback Promotion。
3. PaaS 修改清单仓库到历史镜像 digest。
4. Argo CD 同步。
5. Agent 上报状态。
6. PaaS 记录回滚审计。
```

## 11. 数据表建议

PaaS 默认数据库为 MySQL：


所有业务表、状态机记录、审计日志、outbox/异步任务表默认使用 MySQL 正式表。运行期不得使用内存仓储或内存快照作为业务持久化方案；历史 `repository_snapshots` 快照表只允许在迁移链中删除，旧快照数据不迁移。迁移 SQL 以 MySQL 8.0 方言为准，不使用 PostgreSQL 专属语法、数据类型或迁移能力。JSON 字段可以使用 MySQL `JSON` 类型，但复杂检索、全文检索和日志分析不应依赖 MySQL JSON 查询，后续交给 Loki、Elasticsearch 或 OpenSearch。

## 12. 安全与审计

安全要求：

- 所有用户请求经过 RBAC。
- GitLab、Jenkins、Argo CD 凭据不暴露给用户。
- 平台 Token 加密或哈希存储。
- 用户 Token 不长期用于后台自动化。
- 自动化操作使用平台 Bot 或服务账号。
- Secret 不允许明文展示，只允许创建、更新和引用。
- PaaS 控制面不保存 kubeconfig。
- Agent token 只允许上报所属 clusterId。
- Agent token 支持轮换。

审计事件：

```text
用户登录
创建租户
创建项目
修改成员权限
创建应用
迁移仓库
触发构建
取消构建
创建 Freight
创建 Promotion
审批或拒绝发布
修改环境配置
修改密钥
回滚
禁用集群
Agent 离线
```

## 13. 测试可用版本范围

当前测试可用版本必须具备：

- 用户、租户、项目、RBAC。
- 创建 Application 时只创建应用记录；应用下 Stage 来自租户 DeliverFlow 模板实时投影；创建 BuildPipeline 时直接填写源码类型和源码地址，Workload 在创建或编辑 Workload 时通过 `pipeline_id` 关联流水线。
- Application 作为业务交付上下文。
- Workload 作为最小可部署单元，一个 Workload 对应一个镜像和一个 Deployment 或 StatefulSet。
- Application 支持动态 Workload 数量。
- 单仓库单 Workload，以及 monorepo 下通过 source_path 创建多个 Workload。
- 已有 GitLab 仓库通过 HTTP 地址登记接入。
- 点击构建时按固定名称创建或更新 Jenkins Job 并触发。
- 实时构建日志。
- BuildArtifact 主镜像记录。
- 构建成功生成 Workload Release 候选，并自动尝试创建完整 Freight。
- 自动 Freight 使用本次构建产物覆盖对应 Workload，其他启用 Workload 使用最近一次成功构建产物补齐；缺少任一启用 Workload 成功产物时保留 Release 候选和版本源变更提示，不创建 Freight。
- 用户仍可手动创建 Freight，且 Freight 必须包含所有启用 Workload。
- FreightItem 支持流水线产物和自定义镜像。
- dev/test/staging/prod Stage。
- Promotion 晋级；构建成功、发布完成或验证通过后，系统只为当前 eligible Stage 自动创建 `auto_publish=false` 的手动发布 Promotion，不直接提交 GitOps desired state。
- 构建开始、构建失败和构建成功都通过应用级 WebSocket 通知 Web Console；构建成功事件完成自动 Freight/Promotion 编排后再广播，前端自动刷新流水线、Freight、Promotion 和 Stage 拓扑状态。
- prod 审批。
- GitLab 清单仓库修改。
- Argo CD GitOps 部署。
- PaaS Agent 状态上报。
- 应用列表、应用详情、应用 Workload、Stage 详情、构建详情、发布晋级页面。

当前测试可用版本暂不纳入：

- 绕过 SourceRepository 登记直接使用外部源码仓库作为运行源。
- 跨 Application 的组合发布编排。
- 用户访问 Jenkins UI。
- 用户访问部署清单仓库。
- 用户访问 Argo CD UI。
- PaaS 直连 Kubernetes API Server。
- 自动跨集群故障迁移。
- 服务网格和复杂流量治理。
- 计费系统。
- Issue/MR/Wiki/CI 历史迁移。

## 15. 验收标准

测试可用版本验收时，必须满足：

- 用户能在 PaaS 中为流水线直接配置 Git 或 SVN 源码地址；Git 地址可以刷新分支列表，SVN 可以填写可选 revision。
- 用户能在 Project 下创建 Application，并在应用详情中基于直接源码地址创建 BuildPipeline，指定 source_path 和 BuildSpec；用户能在创建或编辑 Workload 时选择关联流水线。
- 用户能在 PaaS 触发 Jenkins 构建并实时查看日志。
- 构建成功后能生成可发布版本。
- 用户能将版本晋级到 dev/test/staging/prod。
- prod 发布必须经过审批。
- PaaS 通过修改部署清单触发 Argo CD 部署。
- PaaS 不直连 Kubernetes API Server。
- 集群内 Agent 能上报 Argo CD 和 Workload 状态。
- 用户能在 PaaS 看到应用 Stage 状态、实例状态、发布结果和失败原因。
- 所有关键操作都有审计日志。

## 16. 默认假设

- PaaS、GitLab、Jenkins、Argo CD 均部署在企业内部网络。
- PaaS 主数据库使用 MySQL 8.0+，存储引擎为 InnoDB，字符集为 utf8mb4。
- GitLab 源码仓库和部署清单仓库可以使用同一 GitLab 实例的不同 Group。
- 每个 Kubernetes 集群独立部署 Argo CD。
- PaaS Agent 可以访问所在集群内 Kubernetes API Server。
- Jenkins 可以访问 GitLab 托管源码仓库和镜像仓库。
- 镜像仓库已经存在。
- 平台已经准备 Spring Boot 和 Tomcat 运行时基础镜像。
- 企业用户可以通过 OIDC 统一身份源登录 PaaS，也可以由平台管理员创建本地用户后使用账号密码登录。
- 生产发布权限只在 PaaS 内控制。
