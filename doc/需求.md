# PaaS 平台设计备忘

本文档汇总本次会话中讨论的 PaaS 平台设计。

## 产品定位

建设企业内部 PaaS 平台，用户只通过 PaaS Web 控制台操作。Jenkins、GitLab 部署清单仓库、Argo CD 和 Kubernetes API 对普通用户隐藏。

核心集成系统：

- GitLab：用于源码仓库和部署清单仓库。
- Jenkins：用于构建。
- Argo CD：用于 GitOps 部署。
- Kubernetes：用于运行时。
- PaaS Agent：部署在每个 Kubernetes 集群内，用于状态上报和受控的集群侧任务。

## 关键架构决策

- PaaS 控制面是唯一面向用户的控制入口。
- PaaS 不直接调用 Kubernetes API Server。
- 每个 Kubernetes 集群都有自己的 Argo CD 实例。
- 每个 Kubernetes 集群运行一个 PaaS Agent。
- PaaS Agent 在集群内读取 Argo CD Application 和工作负载状态，然后主动上报给 PaaS 控制面。
- 用户看不到 Jenkins UI、GitLab 部署清单仓库、Argo CD UI 或 kubeconfig。
- 源码仓库默认由平台托管在 GitLab 中。
- 现有源码仓库可以迁移到平台托管仓库，但不直接作为运行时源码使用。
- 创建 Application 时不创建源码仓库，也不在创建应用流程中绑定源码仓库。SourceRepository 是 Project 下的独立资源，用户在创建 BuildPipeline 时选择已有的平台托管 SourceRepository。
- Application 表示一个业务交付上下文，可以包含一个或多个 Workload。
- Workload 是最小可独立部署单元，一个 Workload 对应一个容器镜像和一个 Kubernetes 工作负载。
- 当前版本要求一个 Freight 至少包含一个 Workload，且创建 Freight 时必须覆盖该 Application 下所有启用的 Workload。
- 一个 SourceRepository 可以被多个 Application 复用，因此同时支持单仓库和 monorepo 源码布局。
- Web Console 所有用户可见文案必须使用中文，不使用英文作为页面标题、按钮、菜单、提示或状态文案。
- 用户登录支持两种途径：企业 OIDC 登录，以及平台内自主创建的本地用户登录。

## Environment、Stage 和 Cluster 的关系

Stage 化交付流中，用户发布视角统一使用 Stage。Stage 是租户级交付阶段，例如 `dev`、`test`、`staging`、`prod`；每个 Application 根据租户 DeliveryFlow 模板生成自己的应用 Stage。

Environment 仍可作为底层兼容的运行时上下文和部署配置承载对象，但不再作为用户发布绑定的核心对象。Cluster 是实际运行资源池。

关系：

```text
Tenant
  └── DeliveryFlowTemplate
        └── Stage
              └── StageClusterBinding
                    └── Cluster

Application
  └── AppStage
        └── WorkloadEnvironmentConfig / manifest values
```

不要使用 Placement 术语。使用：

- StageClusterBinding（或在兼容旧模块时明确 EnvironmentClusterBinding 的语义已调整为租户 Stage 级绑定）
- stage_cluster_bindings
- ClusterBindingPolicy
- pending_cluster_binding
- cluster_bound

规则：

- Stage key 创建后禁止修改；删除 Stage 会物理删除模板项和该 Stage 的集群池绑定，历史发布记录保留。
- 集群绑定到租户级 Stage 名称，而不是绑定到单个 Application 的某个环境。
- 一个 Stage 可以绑定多个 Cluster；一个 Cluster 也可以被多个 Stage 绑定。
- 发布时从目标 Stage 的集群池中选择目标集群子集，并可覆盖 Namespace；Namespace 默认使用项目名称。
- 审批动作作用于 Freight，入口在 Freight 卡片；验证动作作用于发布后的 Stage，入口在 Stage 卡片；二者均使用弹窗。

## Application、Workload 和仓库粒度

Application 是业务交付上下文，Workload 是平台的最小部署单元：

```text
Application = 一个业务应用或交付上下文
Workload = 一个容器镜像 + 一个 Kubernetes 工作负载
Freight = 一组 Workload 镜像版本的完整发布包
```

示例：

```text
Project: payment
  └── Application: payment
        ├── Workload: admin-web       type=Deployment
        ├── Workload: user-api        type=Deployment
        └── Workload: order-worker    type=StatefulSet
```

如果用户描述的是由前端、后端服务、worker 共同组成的业务应用，可以建模为一个 Application 下的多个 Workload。每个 Workload 仍然独立配置端口、资源、探针、域名、配置文件和可写目录。

仓库策略：

```text
Single repo:
  Project payment
    ├── SourceRepository user-service
    └── Application user-service
          └── BuildPipeline -> SourceRepository user-service

Monorepo:
  Project commerce
    ├── SourceRepository commerce-platform
    └── Application payment
          ├── Workload user-api
          │     └── BuildPipeline source_path=services/user-api
          ├── Workload order-api
          │     └── BuildPipeline source_path=services/order-api
          └── Workload admin-web
                └── BuildPipeline source_path=frontend/admin-web
```

规则：

- 一个 SourceRepository 属于 Project，可以支撑同一 Project 下的多个 Application。
- 一个 Application 可以包含多个 Workload，且 Workload 数量由用户动态增删。
- 一个 Workload 至少有一个镜像来源，可以来自流水线产物，也可以由用户输入自定义镜像。
- Workload 支持 `Deployment` 和 `StatefulSet` 两种工作负载类型。
- BuildPipeline 绑定一个 Workload，并选择一个或多个代码源。代码源保存仓库、源码路径、默认 ref 和构建规格 BuildSpec。当前测试可用版本的 BuildSpec 支持 Java 构建，包含 `build_command`、`artifact_copy_command`、`runtime_base_image`、`artifact_deploy_path` 和 `default_ref`；其中运行时相关字段由构建流水线选择的运行时环境快照填充。
- 创建 Application 时不选择源码仓库、BuildSpec 或运行时环境；用户在创建 BuildPipeline 时选择源码仓库、Workload、构建环境和一个或多个运行时环境。多代码源流水线只在流水线级展示一次运行时环境配置，首个运行时环境作为主产物镜像目标。
- 一个 Build 可以检出多个代码源，并从多个代码源生成多个 BuildArtifact；默认 `is_primary=true` 的产物用于列表和兼容单镜像部署展示。
- Release 表示某个 Workload 的一次可发布镜像版本候选，来源可以是 BuildArtifact。
- Freight 属于一个 Application，由多个 FreightItem 组成。
- 每个 FreightItem 必须绑定一个 Workload 和一个镜像版本。
- 创建 Freight 时，系统必须自动列出 Application 下所有启用 Workload；每个启用 Workload 必须且只能选择一个 FreightItem；禁用或删除 Workload 不进入新 Freight。
- FreightItem 支持 `pipeline_artifact` 和 `custom_image` 两类来源。自定义镜像可以使用 tag 或 digest，但 UI 和审计应提示 tag 可能漂移。
- Stage/Environment 的差异由 WorkloadEnvironmentConfig 或环境 values 承载，Freight 只承载要发布的镜像版本组合。

## 默认环境行为

创建应用时，PaaS 应根据环境模板创建默认环境。

当前测试可用版本默认环境：

```text
dev
test
staging
prod
```

如果尚未注册任何集群：

- 仍然创建环境。
- 环境状态为 `pending_cluster_binding`。
- 不创建 EnvironmentClusterBinding。
- 不创建 Argo CD Application。
- 构建可以运行。
- 部署会被阻止，直到存在集群绑定。

集群注册后：

```text
1. 管理员注册 Cluster。
2. PaaS 扫描 pending_cluster_binding 环境。
3. PaaS 手动或通过 ClusterBindingPolicy 创建 EnvironmentClusterBinding。
4. PaaS 创建 namespace、部署清单路径和 Argo CD Application。
5. Environment 变为可部署。
```

## GitLab 策略

使用平台托管源码仓库。

映射关系：

```text
Tenant      -> GitLab Group
Project     -> GitLab Subgroup
SourceRepository -> GitLab Project
```

推荐 GitLab 目录树：

```text
paas-managed
  └── tenant-a
        └── payment
              └── user-service
```

PaaS 角色到 GitLab 角色的映射：

```text
tenant_owner  -> Owner
project_admin -> Maintainer
developer     -> Developer
viewer        -> Reporter
```

GitLab 权限只表示源码权限。构建、部署和生产审批仍由 PaaS 控制。

SourceRepository 创建和迁移是独立流程，作为 Project 资源在项目详情中管理。Application 创建流程只创建应用记录和默认环境，不选择源码仓库。用户后续在应用详情中创建 BuildPipeline 时选择一个或多个已有 SourceRepository，并为每个代码源填写源码路径、默认 ref 和 BuildSpec。对于单仓应用，通常只有一个代码源。对于多仓构建场景，一个 BuildPipeline 可以同时绑定多个代码源。对于 monorepo 应用，多个 Application 或 Workload 可以通过不同的 source_path 引用同一个 SourceRepository。

## Jenkins 构建配置

Jenkins 对用户隐藏。

PaaS 创建并管理 Jenkins Job。Application 创建时不创建 Jenkins Job；用户点击构建时，PaaS 使用固定名称确保对应 Jenkins Job：

```text
paas/{tenant_name}/{project_name}/{app_name}
```

PaaS 每次触发构建前都会按当前构建配置渲染 Jenkins Job。Job 不存在时创建，已存在时更新配置，然后触发 Jenkins 构建；不再计算 md5 指纹，也不再通过 md5 后缀创建新 Job。

当前测试可用版本由 PaaS 在创建或编辑 BuildPipeline 时保存 BuildSpec，不要求 Jenkins 自己适配所有源码结构；Jenkins 只执行平台统一模板。

BuildSpec 字段：

```text
source_path
build_command
artifact_copy_command
runtime_base_image
artifact_deploy_path
default_ref
```

约束：

- Java 构建由用户填写的 BuildSpec 与构建环境镜像表达，不再在 BuildSpec 中保存构建类型或语言版本字段。
- 用户填写构建命令，但命令只允许在 `source_path` 下执行。
- 镜像生成统一使用平台维护的运行时基础镜像，`runtime_base_image` 必须来自创建 BuildPipeline 时选择的运行时环境。
- `artifact_deploy_path` 是可选产物放置路径；填写时必须是绝对路径，且不允许包含 `..`。
- 在创建流水线页面，`runtime_base_image` 和 `artifact_deploy_path` 来自流水线级运行时环境配置，不随代码源数量变化。
- `dockerfile_path` 是运行时环境管理配置，只在平台管理员的运行时环境管理页维护；创建应用和触发构建时不向用户展示。
- 当前测试可用版本暂不支持客户仓库自带 Jenkinsfile。
- 当前测试可用版本不要求客户在源码仓库提供 Dockerfile；Dockerfile 由平台统一维护在平台专用 Git 仓库中。Jenkins 构建时检出平台 Dockerfile 仓库，Spring Boot jar 默认使用 `java/jar/Dockerfile`，Tomcat war 使用 `java/tomcat/Dockerfile`，并通过 `docker buildx` 构建多架构镜像。

构建日志：

- PaaS 使用 Jenkins progressiveText API。
- 前端通过 SSE 或 WebSocket 从 PaaS 流式接收日志。
- 用户只在 PaaS 中查看日志。

完整构建适配规则见 `doc/Java构建适配方案.md`。

## GitOps 部署策略

PaaS 更新 GitLab 部署清单仓库来表达期望状态。

Argo CD 从 Git 同步到 Kubernetes。

回滚同样通过修改 Git 中的期望状态完成，而不是直接变更集群。

平台提供默认 Workload Helm chart，但共享基础模板只读。每个 Workload 使用一份标准 chart values 表达 Deployment/StatefulSet、镜像、端口、资源、探针、域名、配置文件、可写目录、initContainer、volumeMount 和安全上下文等部署细节。不同环境的差异写入该 Workload 对应环境的 values，不写入 Freight。用户不直接访问 GitLab 部署清单仓库，配置变更由 PaaS 校验、审计并写入清单仓库。

## PaaS Agent

Agent 职责：

- 心跳。
- 集群资源快照。
- 监听 Argo CD Application。
- 监听 Deployment/StatefulSet/ReplicaSet/Pod/Event。
- 向 PaaS 上报标准化后的应用状态。
- 执行受控任务，例如 Argo CD refresh/sync。

Agent API：

```text
POST /api/agent/v1/heartbeat
POST /api/agent/v1/status/report
POST /api/agent/v1/events/report
GET  /api/agent/v1/tasks
POST /api/agent/v1/tasks/result
```

## 前端 UX

前端展示要求：

- 所有用户可见文案必须使用中文，包括导航、按钮、表单标签、校验提示、状态、空状态、错误提示和确认弹窗。
- 技术枚举、API 字段、日志原文和代码片段可以保留英文，例如 `java_springboot`、`build_command`、`source_path`。
- 登录页必须支持本地账号密码登录和 OIDC 登录两种入口。

主要用户概念：

- Application
- Workload
- Environment
- Version/Freight
- Build
- Promotion
- Deployment status

测试可用版本页面：

- 登录
- 项目列表
- 应用列表
- 创建应用向导
- 应用总览
- 应用 Workload
- 环境详情
- 构建列表/详情，包含实时日志
- 晋级页面
- 审计日志

应用详情交互：

- 应用详情主要保留 `应用 Workload` 和 `发布晋级` 两个业务 tab。
- 创建 Workload、编辑部署配置、创建 Freight 均使用弹窗或抽屉承载。
- `发布晋级` tab 内展示按创建时间从左到右排列的 Freight 时间轴。
- 每个 Stage 卡片上提供 `发布` 按钮；点击后可发布 Freight 点亮，用户选择 Freight 后发起 Promotion。
