# 需求文档

## 简介

PaaS 平台是一个企业内部应用交付平台，旨在将源码托管、构建、发布、环境管理、运行状态查看、审批和回滚统一到一个 Web 控制台中，让用户无需直接访问 GitLab、Jenkins、Argo CD、Kubernetes 或部署清单仓库即可完成应用交付。

系统由以下主要组件构成：

- **PaaS Web Console**：面向用户的中文控制台，提供登录、租户项目管理、源码仓库、应用、构建、发布、环境、审批、审计和状态查看等页面。
- **PaaS Control Plane**：平台控制面，负责身份权限、业务编排、领域数据、外部系统适配和 Agent 网关。
- **PaaS Agent**：部署在每个 Kubernetes 集群内，负责采集 Argo CD、工作负载、Pod 和事件状态，并执行受控集群侧任务。
- **GitLab**：承载平台托管源码仓库和部署清单仓库。
- **Jenkins**：执行平台托管的统一构建流水线。
- **Argo CD**：在每个集群内根据 GitOps 期望状态完成部署同步。

平台的核心原则是 PaaS 控制面作为唯一用户入口，普通用户不直接访问 Jenkins、Argo CD、部署清单仓库或 kubeconfig；PaaS 控制面不直接调用 Kubernetes API Server；每个 Kubernetes 集群内独立部署 Argo CD 和 PaaS Agent。

## 词汇表

- **System**：PaaS 平台整体系统。
- **Web Console**：PaaS Web 控制台，用户唯一可见操作入口。
- **Control Plane**：PaaS 控制面，负责编排平台业务流程和外部系统适配。
- **PaaS Agent**：集群内代理，负责状态采集、事件上报和受控任务执行。
- **User**：平台内统一登录主体。
- **Identity**：用户登录身份来源，当前支持 `local` 和 `oidc`。
- **Tenant**：平台最高资源隔离单位。
- **Project**：租户下的应用分组和权限边界。
- **Role**：权限集合。
- **Permission**：权限点，使用 `resource:action` 格式。
- **RoleBinding**：主体在指定作用域下拥有某个角色的授权关系。
- **Group**：用户组，用于批量授权。
- **ServiceAccount**：服务账号，用于系统间调用。
- **AccessToken**：访问令牌。
- **SourceRepository**：平台托管源码仓库。
- **Application**：最小可独立构建、部署、扩缩容和回滚的交付单元。
- **ApplicationSource**：Application 的代码源配置，关联 SourceRepository、源码路径、默认引用和 BuildSpec。
- **BuildSpec**：平台固化的构建规格，描述构建命令、产物收集命令、运行时镜像和默认引用。
- **BuildPipeline**：由 PaaS 渲染并托管到 Jenkins 的构建流水线。
- **BuildRun**：一次构建执行记录。
- **BuildArtifact**：构建产物，当前测试可用版本默认主产物为容器镜像。
- **Environment**：Application 的运行时上下文，例如 `dev`、`test`、`staging`、`prod`。
- **Cluster**：Kubernetes 运行资源池。
- **EnvironmentClusterBinding**：Environment 与 Cluster 的绑定关系。
- **Release**：Application 的可发布版本。
- **Freight**：可在交付流中晋级的版本货物。
- **FreightItem**：Freight 内的产物条目。
- **DeliveryFlow**：Application 的交付流。
- **DeliveryStage**：DeliveryFlow 中指向 Environment 的阶段。
- **Promotion**：将 Freight 晋级到目标环境的发布请求。
- **PromotionApproval**：Promotion 的审批记录。
- **Deployment**：一次部署执行记录。
- **ManifestRevision**：部署清单仓库中的期望状态变更版本。
- **AuditLog**：审计日志。
- **Notification**：平台事件通知。
- **OIDC**：企业统一身份认证协议。
- **Jenkins**：构建执行系统，不作为普通用户入口。
- **GitLab**：源码仓库和部署清单仓库承载系统。
- **Argo CD**：GitOps 同步系统，不作为普通用户入口。

## 需求

### 需求 1：统一入口与底层系统隐藏

**用户故事：** 作为平台用户，我希望只通过 PaaS 控制台完成应用交付操作，以便不需要理解或直接访问 Jenkins、Argo CD、部署清单仓库和 Kubernetes。

#### 验收标准

1. THE System SHALL 将 PaaS Web Console 作为普通用户的唯一交互入口。
2. THE System SHALL 禁止普通用户通过 PaaS 页面跳转到 Jenkins UI、Argo CD UI、部署清单仓库或 kubeconfig 下载入口。
3. WHEN 用户触发构建、发布、回滚或状态查看时，THE Control Plane SHALL 代表用户编排底层系统操作并返回平台领域状态。
4. THE Control Plane SHALL NOT 直接调用 Kubernetes API Server。
5. WHEN 需要获取集群运行状态时，THE System SHALL 通过集群内 PaaS Agent 上报的数据展示状态。
6. IF 外部系统调用失败，THEN THE System SHALL 在 PaaS 中展示中文失败原因并记录审计日志。

---

### 需求 2：用户、身份和登录

**用户故事：** 作为平台用户，我希望既可以使用企业 OIDC 登录，也可以使用平台创建的本地账号密码登录，以便适配不同使用场景。

#### 验收标准

1. THE System SHALL 支持本地账号密码登录和 OIDC 登录两种入口。
2. WHEN 管理员创建本地用户时，THE System SHALL 只保存密码哈希，不保存明文密码。
3. WHEN 本地用户提交账号和密码时，THE System SHALL 校验本地 Identity 和密码哈希，并在成功后签发访问令牌和刷新令牌。
4. WHEN 用户通过 OIDC 登录时，THE System SHALL 校验 `code`、`state`、`nonce` 和 `id_token`，并通过 `issuer` + `subject` 唯一映射到平台 User。
5. WHEN OIDC 用户首次登录且策略允许自动创建时，THE System SHALL 创建 User 和对应 Identity。
6. THE System SHALL 支持同一个 User 后续绑定多个 Identity，但当前测试可用版本至少支持一个本地身份或一个 OIDC 身份。
7. THE Web Console SHALL 在登录页使用中文展示本地登录和 OIDC 登录入口、错误提示和状态文案。

---

### 需求 3：租户、项目和 RBAC

**用户故事：** 作为平台管理员，我希望用租户、项目、角色和权限管理平台资源访问，以便不同团队只能操作被授权的资源。

#### 验收标准

1. THE System SHALL 使用 Tenant 作为最高资源隔离单位。
2. THE System SHALL 使用 Project 作为租户下的应用分组和权限边界。
3. THE System SHALL 支持 User、Group 和 ServiceAccount 作为 RoleBinding 主体。
4. THE System SHALL 支持 `platform`、`tenant`、`project`、`application` 和 `environment` 作用域的权限判定。
5. WHEN 用户访问受保护资源时，THE System SHALL 合并用户直接 RoleBinding 和用户所属 Group 的 RoleBinding 后判断权限。
6. WHEN 用户执行生产发布、审批、回滚或审计查询时，THE System SHALL 执行对应的权限、环境策略和审批策略检查。
7. WHEN 租户、项目、成员或 RBAC 配置发生变更时，THE System SHALL 记录审计日志。

---

### 需求 4：平台托管源码仓库

**用户故事：** 作为开发人员，我希望在 PaaS 中创建或迁移源码仓库，以便源码权限和应用交付流程由平台统一管理。

#### 验收标准

1. THE System SHALL 支持在 PaaS 中创建平台托管 SourceRepository。
2. THE System SHALL 支持将已有源码仓库迁移到平台托管 SourceRepository。
3. THE System SHALL 将 Tenant 映射为 GitLab Group，将 Project 映射为 GitLab Subgroup，将 SourceRepository 映射为 GitLab Project。
4. WHEN SourceRepository 创建或迁移完成时，THE System SHALL 根据 PaaS 角色同步 GitLab 成员权限。
5. THE System SHALL NOT 将外部源码仓库直接作为当前测试可用版本的运行源。
6. WHEN 源码仓库创建、迁移或权限同步失败时，THE System SHALL 展示中文失败原因，并允许任务进入可追踪的失败状态。
7. WHEN SourceRepository 发生创建、迁移、删除或权限同步时，THE System SHALL 记录审计日志。

---

### 需求 5：Application、ApplicationSource 和 BuildSpec 建模

**用户故事：** 作为开发人员，我希望基于已有平台托管仓库创建 Application 并固化构建规格，以便每个可独立交付组件都能被独立构建和发布。

#### 验收标准

1. THE System SHALL 将 Application 建模为最小可独立构建、部署、扩缩容和回滚的交付单元。
2. THE System SHALL NOT 默认将一个 Application 表示为包含多个可独立发布微服务的完整业务系统。
3. WHEN 用户创建 Application 时，THE System SHALL 要求用户选择已有 SourceRepository。
4. THE System SHALL 支持一个 SourceRepository 被多个 Application 复用。
5. THE System SHALL 支持 ApplicationSource 保存仓库、源码路径、默认 ref 和 BuildSpec。
6. THE System SHALL 支持 `java_springboot`、`java_tomcat` 和 `node_static` 构建适配类型；其中当前 Java 构建按平台统一模板执行。
7. WHEN 用户创建 BuildPipeline 并填写 BuildSpec 时，THE System SHALL 校验源码路径、构建命令、产物拷贝命令、运行时环境和产物部署路径。
8. IF BuildSpec 不合法，THEN THE System SHALL 阻止创建或更新 Application，并返回中文校验提示。

---

### 需求 6：Environment、Cluster 和环境绑定

**用户故事：** 作为平台管理员和应用负责人，我希望应用创建后自动具备标准环境，并能把环境绑定到合适集群，以便发布流程有明确运行目标。

#### 验收标准

1. WHEN Application 创建成功时，THE System SHALL 自动创建 `dev`、`test`、`staging` 和 `prod` 默认环境。
2. THE System SHALL 使用 Environment 表示运行时上下文，使用 Cluster 表示实际运行资源池。
3. THE System SHALL 使用 EnvironmentClusterBinding 表示 Environment 与 Cluster 的绑定关系。
4. IF 创建 Application 时尚无可用集群，THEN THE System SHALL 创建环境并将环境状态设为 `pending_cluster_binding`。
5. WHEN 环境处于 `pending_cluster_binding` 时，THE System SHALL 允许构建执行，但阻止部署和 Promotion。
6. WHEN 管理员注册 Cluster 后，THE System SHALL 支持手动或按 ClusterBindingPolicy 为待绑定环境创建 EnvironmentClusterBinding。
7. THE System SHALL NOT 使用文档外的环境调度术语替代 EnvironmentClusterBinding。

---

### 需求 7：Jenkins 托管构建

**用户故事：** 作为开发人员，我希望在 PaaS 中触发 Jenkins 构建，以便使用统一模板生成可发布镜像而不需要维护 Jenkinsfile。

#### 验收标准

1. THE System SHALL 由 PaaS 创建和管理 Jenkins Job，普通用户不直接维护 Jenkinsfile。
2. WHEN 用户点击构建时，THE System SHALL 使用固定 Job 名称 `paas/{tenant_name}/{project_name}/{app_name}` 创建或更新 Jenkins Job。
3. WHEN 每次触发构建前，THE System SHALL 按本次 BuildRun 的 ApplicationSource、BuildSpec、源码 ref、运行时环境、镜像目标和回调地址重新渲染 Jenkinsfile 并更新 Jenkins Job。
4. THE System SHALL NOT 依赖 Jenkins Job 配置 md5 指纹决定是否更新 Job。
5. THE Jenkins 模板 SHALL 首次 clone 源码和 Dockerfile 仓库，后续复用工作树并执行 `git fetch --prune --tags origin`、解析 ref、`checkout --detach`、`reset --hard` 和 `clean -fdx` 以获取最新变化。
6. THE Jenkins 模板 SHALL 在每个 ApplicationSource 的 `source_path` 内执行受限构建命令，并把产物收集到 `$PAAS_ARTIFACT_OUTPUT`。
7. THE Jenkins 模板 SHALL 使用平台维护的 Dockerfile 和运行时基础镜像构建容器镜像，不使用客户源码仓库内 Dockerfile 作为当前测试可用版本的镜像构建入口。

---

### 需求 8：构建日志、镜像标签和构建产物

**用户故事：** 作为开发人员，我希望在 PaaS 中查看构建进度、构建日志和产物信息，以便判断构建是否可发布。

#### 验收标准

1. WHEN Jenkins 构建开始、运行、成功、失败、取消或 unstable 时，THE System SHALL 更新 BuildRun 状态。
2. THE System SHALL 支持在 PaaS 中持续读取构建增量日志。
3. WHEN 构建日志过长时，THE Web Console SHALL 默认展示截断后的日志，并仅在用户点击 `FullText` 后展示完整日志。
4. THE System SHALL 在展示构建日志、错误响应和审计日志前脱敏 token、password、secret 和 registry password。
5. WHEN 构建成功时，THE System SHALL 生成 BuildArtifact，并记录镜像地址、digest、commit、source_key 和主产物标识。
6. THE System SHALL 使用 `yyyyMMdd-{commit}-...` 格式生成应用镜像 tag，且 tag SHALL NOT 包含流水线 id。
7. IF 构建失败、取消或 unstable，THEN THE System SHALL NOT 生成可发布 Release。
8. IF 产物拷贝命令没有向 `$PAAS_ARTIFACT_OUTPUT` 写入任何文件，THEN THE BuildRun SHALL 失败并展示中文原因。

---

### 需求 9：Release、Freight 和 DeliveryFlow

**用户故事：** 作为发布负责人，我希望构建成功后自动形成可晋级版本，以便同一个版本能按交付流进入不同环境。

#### 验收标准

1. WHEN BuildRun 成功并产生有效 BuildArtifact 时，THE System SHALL 创建 Release。
2. WHEN Release 创建成功时，THE System SHALL 创建包含 FreightItem 的 Freight。
3. THE System SHALL 支持一个 Freight 包含一个或多个 FreightItem，以保留未来多应用协同晋级能力。
4. WHEN Application 创建成功时，THE System SHALL 创建默认 DeliveryFlow。
5. THE DeliveryFlow SHALL 默认包含指向 `dev`、`test`、`staging` 和 `prod` Environment 的 DeliveryStage。
6. THE System SHALL 在版本列表中展示 Release、Freight、构建来源、commit、镜像和当前晋级状态。
7. IF BuildArtifact 缺少镜像 digest 或 commit 信息，THEN THE System SHALL 阻止生成可发布 Release。

---

### 需求 10：Promotion、审批和回滚

**用户故事：** 作为发布负责人，我希望将 Freight 按环境晋级，并在生产环境执行审批和回滚控制，以便发布过程可控可审计。

#### 验收标准

1. THE System SHALL 支持将 Freight 晋级到 `dev`、`test`、`staging` 和 `prod` 环境。
2. WHEN 目标环境为 `prod` 且策略要求审批时，THE System SHALL 创建 PromotionApproval 并等待审批通过后再部署。
3. THE System SHALL 禁止生产环境 Promotion 发起人自审批。
4. WHEN 审批被拒绝时，THE Promotion SHALL 进入拒绝状态，并且 SHALL NOT 修改生产环境期望状态。
5. WHEN Promotion 被批准时，THE System SHALL 创建 Deployment 并触发 GitOps 期望状态变更。
6. THE System SHALL 支持基于历史 Freight 发起回滚 Promotion。
7. WHEN 执行回滚时，THE System SHALL 通过修改 Git 中的期望状态完成回滚，而不是直接操作 Kubernetes。
8. WHEN Promotion 创建、审批、拒绝、中止、部署或回滚时，THE System SHALL 记录审计日志。

---

### 需求 11：GitOps 部署与部署模板

**用户故事：** 作为平台管理员，我希望平台通过 GitOps 管理部署期望状态，以便发布过程可追踪、可回放且不直连集群 API。

#### 验收标准

1. THE System SHALL 通过 GitLab 部署清单仓库保存每个环境的期望状态。
2. WHEN EnvironmentClusterBinding 创建完成时，THE System SHALL 创建 namespace、部署清单路径和 Argo CD Application 清单所需数据。
3. WHEN Promotion 进入部署阶段时，THE System SHALL 修改部署清单仓库并创建 ManifestRevision。
4. THE System SHALL 根据环境策略决定直接提交或创建合并请求。
5. WHEN GitLab 部署清单提交或合并请求创建失败时，THE Deployment SHALL 进入失败状态并展示中文原因。
6. THE System SHALL 由集群内 Argo CD 监听部署清单仓库并同步到 Kubernetes。
7. THE Control Plane SHALL NOT 使用 kubeconfig 或 Kubernetes API Server 直接创建、更新或删除工作负载。
8. WHEN 部署模板发生变更时，THE System SHALL 记录审计日志。

---

### 需求 12：PaaS Agent 与集群状态上报

**用户故事：** 作为应用负责人，我希望在 PaaS 中看到真实运行状态和失败原因，以便不进入集群也能判断应用健康情况。

#### 验收标准

1. THE System SHALL 要求每个 Kubernetes 集群内部署独立 PaaS Agent。
2. THE PaaS Agent SHALL 主动连接 Control Plane 并上报心跳。
3. THE PaaS Agent SHALL 采集 Argo CD Application 的 sync status、health status 和 operation state。
4. THE PaaS Agent SHALL 采集必要的 Workload、Pod 和 Event 状态，并转换为 PaaS 标准状态模型。
5. WHEN PaaS Agent 离线或心跳超时，THE System SHALL 将 Cluster 标记为离线或异常。
6. WHEN Argo CD Application Degraded 或 Pod 出现 ImagePullBackOff 等异常时，THE System SHALL 在 PaaS 中展示环境状态、实例状态和失败原因。
7. WHEN Control Plane 下发受控任务时，THE PaaS Agent SHALL 拉取任务、执行允许的集群侧操作并回传 ClusterTaskResult。

---

### 需求 13：Web Console 用户体验

**用户故事：** 作为平台用户，我希望 PaaS 控制台的页面、表单、状态和错误提示都清晰一致，以便高效完成日常交付操作。

#### 验收标准

1. THE Web Console SHALL 使用中文展示所有用户可见的页面标题、菜单、按钮、表单标签、校验提示、状态、空状态和错误提示。
2. THE Web Console SHALL 提供登录、项目列表、应用列表、创建应用向导、应用详情、环境详情、构建详情、发布晋级、审批、回滚和审计日志页面。
3. WHEN 用户创建 BuildPipeline 时，THE Web Console SHALL 引导用户选择 SourceRepository、源码路径、构建环境、构建命令、产物拷贝命令和运行时环境。
4. WHEN 用户查看应用详情时，THE Web Console SHALL 展示环境状态、最新构建、最新 Release、当前 Freight 和部署结果。
5. WHEN 用户查看发布晋级页面时，THE Web Console SHALL 展示 Freight、目标环境、审批状态、部署状态和失败原因。
6. THE Web Console SHALL 通过 API Server 访问后端能力，并 SHALL NOT 直接访问 GitLab、Jenkins、Argo CD 或 Kubernetes。
7. WHEN API 返回失败时，THE Web Console SHALL 展示可理解的中文错误信息，并保留用户可重试的上下文。

---

### 需求 14：审计、安全和凭据保护

**用户故事：** 作为安全审计人员，我希望平台关键操作可追溯且凭据不泄露，以便满足企业内部安全要求。

#### 验收标准

1. THE System SHALL 对本地用户创建、密码重置、登录、Token 签发、OIDC 登录和 Identity 绑定记录审计日志。
2. THE System SHALL 对租户、项目、成员、RBAC、SourceRepository、Application、BuildSpec、环境配置、Promotion、审批、回滚和部署模板变更记录审计日志。
3. THE System SHALL NOT 在 API 响应中返回 password_hash、token_hash、client_secret、kubeconfig 或外部系统凭据。
4. THE System SHALL NOT 在构建日志、审计日志或错误响应中暴露 token、password、secret、registry password 或 kubeconfig。
5. WHEN Agent token 或外部系统凭据轮换时，THE System SHALL 记录关键审计或运行日志。
6. THE System SHALL 使用受控凭据向 GitLab、Jenkins、镜像仓库和 OIDC Provider 发起调用。
7. WHEN 用户无权限访问资源时，THE System SHALL 返回中文拒绝原因，并且 SHALL NOT 泄露目标资源敏感细节。

### 需求 15：通知与事件

**用户故事：** 作为发布负责人和平台管理员，我希望关键构建、发布、审批和集群事件能被通知到相关人员，以便及时处理阻塞和失败。

#### 验收标准

1. THE System SHALL 为构建完成、构建失败、Promotion 待审批、审批通过、审批拒绝、部署失败、回滚完成和 Agent 离线等事件生成平台事件。
2. THE System SHALL 支持按租户、项目、应用和环境策略配置通知接收人。
3. WHEN 生产发布进入待审批状态时，THE System SHALL 通知具备审批权限的用户。
4. WHEN 构建或部署失败时，THE System SHALL 通知触发人和相关项目负责人。
5. WHEN 通知发送失败时，THE System SHALL 记录失败原因并允许后续重试或人工排查。
6. THE Notification SHALL NOT 包含明文凭据、token、secret 或 kubeconfig。

### 需求 16：外部系统适配与可测试性

**用户故事：** 作为平台开发人员，我希望外部系统调用被清晰隔离并可替换测试，以便平台模块能独立开发、独立验证和稳定演进。

#### 验收标准

1. THE System SHALL 通过 adapter 或 port interface 隔离 GitLab、Jenkins、Argo CD、镜像仓库和 Agent 网关调用。
2. THE identity-access、tenant-project、source-repository、application-environment、build、release-delivery、gitops-deployment、cluster-agent、audit 和 notification 模块 SHALL 只通过公开接口、事件或明确数据契约协作。
3. THE build 模块 SHALL NOT 直接查询 Application 模块内部数据表，而应通过 ApplicationQueryPort 获取所需 ApplicationSource 和 BuildSpec。
4. THE delivery 模块 SHALL NOT 直接修改 BuildRun 内部数据，而应通过 BuildQueryPort 获取可发布 BuildArtifact。
5. THE cluster-agent 模块 SHALL 通过 ClusterCommandPort 或事件更新集群相关状态。
6. THE System SHALL 支持使用 mock、fake adapter 或测试桩独立测试各模块。
7. THE System SHALL 提供主链路、失败场景、审计安全和前端交互的端到端验收能力。
