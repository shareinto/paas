# paas-server 和 paas-agent 配置样例

本文档说明测试可用版 `paas-server` 和 `paas-agent` 的启动配置。配置样例位于：

- `deploy/config/paas-server.env.example`
- `deploy/config/paas-agent.env.example`

样例文件只能保存占位值，不应写入真实密码、token、secret、kubeconfig 或生产地址。

## paas-server

当前 `paas-server` 通过环境变量读取启动配置：

```bash
set -a
source deploy/config/paas-server.env.example
set +a
go run ./cmd/paas-server
```

配置项：

| 环境变量 | 默认值 | 说明 |
| --- | --- | --- |
| `PAAS_HTTP_ADDR` | `:8080` | 控制面 HTTP 监听地址。 |
| `PAAS_AUTO_MIGRATE` | `false` | 设置为 `true` 时，启动前执行 MySQL 迁移；只执行未记录版本，不会重放已执行过的 migration。 |
| `PAAS_LOCAL_REGISTRATION_ENABLED` | `true` | 是否开放本地账号自助注册。默认开放；试点或生产环境如需仅允许管理员创建本地用户，可设置为 `false`。 |
| `PAAS_REPOSITORY_DRIVER` | `mysql` | 兼容旧部署脚本的保留变量。业务仓储固定使用 MySQL 正式表，不支持内存仓储。 |
| `MYSQL_HOST` | `127.0.0.1` | MySQL 地址。 |
| `MYSQL_PORT` | `3306` | MySQL 端口。 |
| `MYSQL_DATABASE` | `paas` | MySQL 数据库名。 |
| `MYSQL_USER` | `paas` | MySQL 用户名。 |
| `MYSQL_PASSWORD` | 空 | MySQL 密码，必须通过环境凭据注入。 |
| `GITLAB_BASE_URL` | 空 | GitLab 地址。未配置时使用 fake adapter。 |
| `GITLAB_TOKEN` | 空 | GitLab 访问 token，必须通过环境凭据注入，不得写入仓库。 |
| `GITLAB_HTTP_URL_ALIASES` | 空 | 逗号分隔的 GitLab HTTP 地址别名。用于 `GITLAB_BASE_URL` 是内网访问地址、用户登记仓库时填写外部访问地址的场景。 |
| `GITLAB_ROOT_GROUP_PATH` | `paas` | 历史兼容配置；当前 SourceRepository 登记不再按该 Group 创建源码仓库。 |
| `GITLAB_WEBHOOK_CALLBACK_URL` | 空 | 历史兼容配置；当前 SourceRepository 登记不再自动配置源码仓库 Webhook。 |
| `GITOPS_GITLAB_BASE_URL` | 空 | 部署清单仓库和平台 Chart 仓库所在 GitLab 地址。留空时兼容使用 `GITLAB_BASE_URL`。 |
| `GITOPS_GITLAB_TOKEN` | 空 | 部署清单仓库和平台 Chart 仓库 GitLab 访问 token。留空时兼容使用 `GITLAB_TOKEN`。 |
| `GITLAB_MANIFEST_PROJECT_ID` | 空 | 部署清单仓库 GitLab Project ID。与 GitOps GitLab 地址、token 和 `GITLAB_CHART_PROJECT_ID` 同时配置后，发布会通过 GitLab Repository API 写入 values 和 Argo CD Application。 |
| `GITLAB_MANIFEST_REPO_URL` | 空 | Argo CD 读取部署清单仓库的 Git URL。留空时按 GitOps GitLab 地址和 Project ID 生成兜底值，生产建议显式配置 SSH 或 HTTPS 地址。 |
| `GITLAB_CHART_PROJECT_ID` | 空 | 平台 Chart 独立仓库 GitLab Project ID。发布时 PaaS 会确保 `charts/{chart}` 文件存在，并创建版本 tag。 |
| `GITLAB_CHART_REPO_URL` | 空 | Argo CD 读取平台 Chart 仓库的 Git URL。留空时按 GitOps GitLab 地址和 Project ID 生成兜底值。 |
| `GITOPS_GITLAB_TIMEOUT_SECONDS` | 空 | 部署清单仓库 GitLab HTTP 调用超时时间。留空时使用 `GITLAB_TIMEOUT_SECONDS`。 |
| `GITOPS_GITLAB_RETRY_MAX` | 空 | 部署清单仓库 GitLab 临时失败重试次数。留空时使用 `GITLAB_RETRY_MAX`。 |
| `GITLAB_TIMEOUT_SECONDS` | `10` | GitLab HTTP 调用超时时间。 |
| `GITLAB_RETRY_MAX` | `2` | GitLab 临时失败重试次数。 |
| `PAAS_PLATFORM_CHART_NAME` | `paas-app` | PaaS 维护的平台标准 Helm Chart 名称。 |
| `PAAS_PLATFORM_CHART_VERSION` | `0.1.0` | PaaS 维护的平台标准 Helm Chart 版本。发布时使用 tag `{chart}-v{version}`，避免线上应用跟随未验证模板变更漂移。 |
| `JENKINS_BASE_URL` | 空 | Jenkins 地址。未配置时使用 fake adapter。 |
| `JENKINS_USERNAME` | 空 | Jenkins API 用户名。 |
| `JENKINS_TOKEN` | 空 | Jenkins API token，必须通过环境凭据注入，不得写入仓库。 |
| `JENKINS_TIMEOUT_SECONDS` | `10` | Jenkins HTTP 调用超时时间。 |
| `JENKINS_RETRY_MAX` | `2` | Jenkins 临时失败重试次数。 |
| `JENKINS_DEFAULT_TEMPLATE_ID` | `java-unified-v1` | 兼容历史 Jenkins 模板的默认 ID，新构建流程优先使用全局构建模板。 |
| `JENKINS_CALLBACK_BASE_URL` | `http://127.0.0.1:8080` | Jenkins 回调 PaaS 控制面的基础地址，构建时会拼接 `/api/builds/{buildRunId}/callback`。 |
| `IMAGE_REPOSITORY` | `registry.local/order-api` | 构建产物镜像仓库前缀，触发构建时追加应用名称。 |
| `DOCKERFILE_REPOSITORY_URL` | `ssh://git@192.168.100.80:2422/paas/dockerfiles.git` | 平台统一 Dockerfile 仓库地址。 |
| `DOCKERFILE_REPOSITORY_REF` | `main` | 平台 Dockerfile 仓库 ref，建议使用 tag 或 commit。 |
| `DOCKERFILE_REPOSITORY_CREDENTIALS_ID` | 空 | Jenkins 中用于检出平台 Dockerfile 仓库的凭据 ID。 |

`PAAS_AUTO_MIGRATE=true` 时，启动流程会读取 `MYSQL_*`，连接 MySQL 并执行 `internal/migrations.All()` 中的迁移。迁移失败时服务不会继续启动。

`PAAS_LOCAL_REGISTRATION_ENABLED` 控制登录页本地账号自助注册入口和公开注册接口。默认值为 `true`，注册成功后会创建本地用户、签发平台 token，但不会自动授予任何 RBAC 角色；后续仍需管理员按租户、项目或平台范围分配权限。设置为 `false` 时，注册接口返回“注册功能未开启”，管理员仍可通过用户管理接口创建本地用户。

开发阶段如果修改了已执行过的 migration 文件，旧开发库不会因为 `PAAS_AUTO_MIGRATE=true` 自动重放该版本。此类 schema 漂移应按 `doc/MySQL迁移与启动流程.md` 的开发库清库重建流程处理。进入试点或生产后，不应再修改已执行 migration，应追加更高版本 migration。

本地开发脚本 `scripts/dev-up.sh` 额外支持 `--recreate-db` 和 `PAAS_DEV_RECREATE_DB=true`。该开关只用于开发阶段，会删除并重建 `MYSQL_DATABASE`，再用当前 migration 全量建表；不应在需要保留数据的环境使用。`--recreate-db` 默认在后端 ready 后通过 `mysql` 客户端直连数据库注入 SBG/MACC 开发测试数据；如需跳过，使用 `--no-seed-dev-data` 或设置 `PAAS_DEV_SEED_DATA=false`。如需不重建但注入测试数据，可使用 `--seed-dev-data`；如需验证控制面 API 写入链路，可单独执行 `scripts/dev-seed.sh`。本机没有 `mysql` 命令时，`scripts/dev-up.sh` 会使用 Docker 镜像 `m.daocloud.io/docker.io/library/mysql:8.0` 作为一次性 MySQL 客户端；可通过 `PAAS_MYSQL_CLIENT_IMAGE` 覆盖。

`paas-server` 使用 MySQL 正式表持久化业务仓储。运行期不再提供内存仓储，也不再使用 `repository_snapshots` 快照表；迁移 `drop_repository_snapshots` 会删除历史快照表，旧快照数据不迁移。

配置 `GITLAB_BASE_URL` 和 `GITLAB_TOKEN` 后，SourceRepository 登记会校验用户填写的 HTTP 仓库地址属于该 GitLab，解析 GitLab Project 并扫描默认分支；平台不再创建源码仓库、保护分支或配置 Webhook。未配置 GitLab 时继续使用 fake adapter，便于本地开发。如果控制面通过 Docker 网络名或内网域名访问 GitLab，而用户登记仓库时使用外部访问地址，应把外部地址加入 `GITLAB_HTTP_URL_ALIASES`，平台仍只通过 `GITLAB_BASE_URL` 调用 GitLab API。

配置 `GITLAB_MANIFEST_PROJECT_ID` 和 `GITLAB_CHART_PROJECT_ID` 后，GitOps 发布会使用 GitLab API 写入两个仓库：平台 Chart 仓库保存版本化 `charts/paas-app`，部署清单仓库保存每个应用、环境、集群的 values 和 Argo CD Application。部署清单仓库和源码仓库不在同一个 GitLab 时，应配置 `GITOPS_GITLAB_BASE_URL` 和 `GITOPS_GITLAB_TOKEN`；如果留空，则兼容使用源码仓库的 `GITLAB_BASE_URL` 和 `GITLAB_TOKEN`。Argo CD Application 使用 multi-source，从 Chart 仓库读取固定 tag，从清单仓库读取 `$values/apps/{app}/{env}/values.yaml`。如果更新 Chart 模板，应升级 `PAAS_PLATFORM_CHART_VERSION` 并发布新 tag，避免所有线上应用自动引用同一个可变模板。

配置 `JENKINS_BASE_URL` 和 `JENKINS_TOKEN` 后，构建模块会调用真实 Jenkins：平台管理员先在 Web Console 的“构建管理”页面维护构建环境、运行时环境和全局构建模板，用户创建 BuildPipeline 时选择已启用构建环境和一个或多个运行时环境。触发构建时 PaaS 会按 `paas/{tenant_name}/{project_name}/{app_name}` 确保 Jenkins Folder/Job，Job 不存在时创建，已存在时按本次 BuildRun 重新渲染 Jenkinsfile 后更新，触发 Jenkins 时不传递构建参数，并通过 progressiveText 拉取日志。Jenkins 会增量检出应用源码和平台 Dockerfile 仓库，Spring Boot jar 使用 `java/jar/Dockerfile`，Tomcat war 使用 `java/tomcat/Dockerfile`，并通过 `docker buildx` 构建多架构镜像。Jenkins 构建节点应提供持久化 `/backup_data` 目录，用于 Maven、Gradle、npm、Yarn、pnpm 依赖缓存和 buildx layer cache；这些缓存默认是节点本地缓存，跨节点调度时不会自动共享。未配置 Jenkins 时继续使用 fake adapter。

## paas-agent

当前 `paas-agent` 通过环境变量读取启动配置，并使用集群内 ServiceAccount 初始化 Kubernetes client。生产和试点环境应通过 `deploy/paas-agent` Helm Chart 部署到 Kubernetes 集群内运行。

配置项：

| 环境变量 | 默认值 | 说明 |
| --- | --- | --- |
| `PAAS_CLUSTER_ID` | 无 | Agent 所属集群 ID，必须与控制面 Cluster 记录一致。 |
| `PAAS_CONTROL_PLANE_URL` | 无 | PaaS 控制面地址，Agent 调用 `/api/agent/v1/*`。 |
| `PAAS_AGENT_TOKEN` | 无 | 控制面生成的 Agent token，应通过 Kubernetes Secret 注入。 |
| `PAAS_AGENT_NAMESPACES` | 空 | 逗号分隔的采集 namespace，例如 `argocd,order-dev,order-test`。 |
| `PAAS_ARGOCD_NAMESPACE` | `argocd` | Argo CD Application 所在 namespace。 |
| `PAAS_HEARTBEAT_INTERVAL` | `10s` | Agent 心跳周期。 |
| `PAAS_SNAPSHOT_INTERVAL` | `30s` | 状态快照上报周期。 |

Agent 请求控制面时会携带：

- `Authorization: Bearer <PAAS_AGENT_TOKEN>`
- `X-PaaS-Cluster-ID: <PAAS_CLUSTER_ID>`

控制面必须校验 token 只能上报所属集群，避免跨集群冒用。

## Kubernetes Secret 注入示例

`PAAS_AGENT_TOKEN` 不应出现在 ConfigMap 或普通 manifest 中。示例：

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: paas-agent-secret
  namespace: paas-system
type: Opaque
stringData:
  PAAS_AGENT_TOKEN: "replace-with-control-plane-issued-token"
```

Deployment 中通过单项 `secretKeyRef` 注入该 Secret。Helm Chart 位于 `deploy/paas-agent/`，会渲染 ServiceAccount、最小 RBAC、ConfigMap、Secret 和 Deployment。

## Kubernetes Helm 部署示例

生产或试点环境建议先手动创建 Secret，再通过 `secret.existingSecret` 引用，避免真实 token 进入 Helm values 或命令历史：

```bash
kubectl create namespace paas-system
kubectl -n paas-system create secret generic paas-agent-secret \
  --from-literal=PAAS_AGENT_TOKEN='replace-with-control-plane-issued-token'
```

构建并推送镜像后安装：

```bash
docker build -f deploy/paas-agent/Dockerfile -t registry.example/paas/paas-agent:0.1.0 .
docker push registry.example/paas/paas-agent:0.1.0

helm upgrade --install paas-agent deploy/paas-agent \
  --namespace paas-system \
  --set image.repository=registry.example/paas/paas-agent \
  --set image.tag=0.1.0 \
  --set config.clusterID=cluster_dev \
  --set config.controlPlaneURL=http://paas-server.paas-system.svc.cluster.local:8080 \
  --set config.agentNamespaces=argocd,order-dev,order-test \
  --set config.argocdNamespace=argocd \
  --set secret.existingSecret=paas-agent-secret
```

测试渲染时可以使用占位 token：

```bash
helm template paas-agent deploy/paas-agent \
  --namespace paas-system \
  --set config.clusterID=cluster_dev \
  --set config.controlPlaneURL=http://paas-server.paas-system.svc.cluster.local:8080 \
  --set secret.token=change-me
```

Agent token 轮换后，应更新 `paas-agent-secret` 并滚动重启 Deployment：

```bash
kubectl -n paas-system rollout restart deployment/paas-agent
```

## 安全约束

- 不在代码、文档或日志中记录真实 token、密码、secret、kubeconfig。
- `.env` 实际配置文件应只保存在部署环境，并设置最小读取权限。
- Web Console 是用户入口；普通用户不直接访问 Jenkins、Argo CD、部署清单仓库或 kubeconfig。
- PaaS 控制面不直接调用 Kubernetes API Server，集群侧状态和受控任务由 `paas-agent` 完成。
