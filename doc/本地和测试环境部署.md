# 本地和测试环境部署说明

本文档用于把当前测试可用版启动为可本地调试、可测试环境验收的版本。配置样例见 `doc/配置样例.md`，MySQL 迁移细节见 `doc/MySQL迁移与启动流程.md`。

## 运行前提

- Go `1.25`，与 `go.mod` 保持一致。
- Node.js 和 npm，前端工程位于 `web/console`。
- 本地和测试环境都需要 MySQL 8.0；`paas-server` 业务仓储固定使用 MySQL 正式表，不提供内存演示数据。
- 每个 Kubernetes 集群内部署独立 Argo CD 和 `paas-agent`，普通用户不直接访问 Argo CD UI、Jenkins UI、部署清单仓库或 kubeconfig。

## 本地开发启动

本地启动默认使用 MySQL 正式表和演示配置，适合验证 API、Web Console 和中文用户流程。推荐使用一键脚本同时启动后端和前端：

```bash
cp deploy/config/paas-dev.env.example deploy/config/paas-dev.env
# 按需修改 deploy/config/paas-dev.env 中的 GitLab、Jenkins、MySQL 和镜像仓库配置
./scripts/dev-up.sh
```

脚本会读取 `deploy/config/paas-dev.env`，启动 `paas-server` 和 Web Console，并自动设置 `VITE_API_BASE_URL`。未配置 `GITLAB_BASE_URL/GITLAB_TOKEN` 或 `JENKINS_BASE_URL/JENKINS_TOKEN` 时，对应集成使用 fake adapter。

默认启动不会重建数据库：

```bash
./scripts/dev-up.sh
```

默认启动也不会注入开发测试数据。

如果需要按当前 migration 全量新建库表，使用显式重建参数：

```bash
./scripts/dev-up.sh --recreate-db
```

`--recreate-db` 会先通过 `mysql` 客户端删除并重建 `MYSQL_DATABASE`，然后强制设置 `PAAS_AUTO_MIGRATE=true` 启动后端，由当前代码的全部 migration 从空库建表。后端 `/readyz` 可用后，脚本会默认通过 `mysql` 客户端直连数据库注入一组控制台验证数据，再启动 Web Console。也可以用环境变量触发同样行为：

执行 `--recreate-db` 前需要确保旧的 `paas-server` 已停止；脚本会在发现当前后端 `/readyz` 已可用时拒绝重建，避免旧进程占用端口导致新后端启动失败。

如果本机未安装 `mysql` 命令，脚本会自动使用 Docker 镜像 `m.daocloud.io/docker.io/library/mysql:8.0` 作为一次性 MySQL 客户端；可通过 `PAAS_MYSQL_CLIENT_IMAGE` 覆盖该镜像。

```bash
PAAS_DEV_RECREATE_DB=true ./scripts/dev-up.sh
```

重建但跳过开发测试数据：

```bash
./scripts/dev-up.sh --recreate-db --no-seed-dev-data
```

不重建数据库但手工注入或更新开发测试数据：

```bash
./scripts/dev-up.sh --seed-dev-data
```

也可以在后端已启动时单独执行：

```bash
./scripts/dev-seed.sh
```

开发测试数据包含租户 `sbg`、项目 `macc`、源码仓库 `log-receiver`、应用 `log-receiver`、Workload `log-receiver` 和构建流水线 `main`。`scripts/dev-up.sh --recreate-db` 和 `scripts/dev-up.sh --seed-dev-data` 会在后端 ready、migration 已完成后，通过 `mysql` 客户端直连数据库插入或更新这批开发数据，不调用业务 API。脚本会将该仓库的 `http_url` 更新为 `http://192.168.100.80/paas/sbg/macc/log-receiver.git`，将 `ssh_url` 更新为 `ssh://git@192.168.100.80:2422/paas/sbg/macc/log-receiver.git`，并将状态置为 `ready`；这是开发种子数据专用行为，不改变业务 API。

独立执行 `scripts/dev-seed.sh` 仍会按该脚本自身逻辑通过控制面 API 注入或更新开发测试数据，主要用于需要验证 API 写入链路的场景。该脚本会通过源码仓库登记 API 提交 `http://192.168.100.80/paas/sbg/macc/log-receiver.git`；平台会按当前 GitLab 配置解析并扫描该仓库，校验失败时种子数据注入失败，不再降级调用仓库迁移 API 或直接修改 MySQL。如果 `GITLAB_BASE_URL` 使用 Docker 网络名或内网地址，例如 `http://gitops`，应将 `http://192.168.100.80` 加入 `GITLAB_HTTP_URL_ALIASES` 后再执行 API 种子脚本。

如果只想启动时执行未应用 migration，但不清库：

```bash
PAAS_AUTO_MIGRATE=true ./scripts/dev-up.sh
```

也可以手动启动后端：

```bash
set -a
source deploy/config/paas-server.env.example
set +a
export PAAS_AUTO_MIGRATE=false
export PAAS_HTTP_ADDR=:8080
go run ./cmd/paas-server
```

健康检查：

```bash
curl -fsS http://127.0.0.1:8080/healthz
```

手动启动 Web Console：

```bash
cd web/console
npm install
VITE_API_BASE_URL=http://127.0.0.1:8080 npm run dev
```

未设置 `VITE_API_BASE_URL` 时，Web Console 使用前端 mock API；设置后会调用真实后端开发 API。

本地演示账号：

```text
username = admin
password = password
```

### 开发库重建约定

当前项目仍处于开发阶段，本地和临时开发验收环境以“当前 migration 全量新建库表”为准。遇到缺字段、缺表或迁移漂移时，标准处理方式是清库重建，而不是为旧开发库状态追加兼容迁移。

清库重建前应停止后端服务，并确认库内应用、构建、Freight、Stage、审计等数据可以丢弃。共享开发库重建前必须通知相关使用者，并按需执行 `mysqldump` 备份。日常开发推荐直接使用 `./scripts/dev-up.sh --recreate-db`，如不需要 SBG/MACC 测试数据则使用 `./scripts/dev-up.sh --recreate-db --no-seed-dev-data`。手工步骤见 `doc/MySQL迁移与启动流程.md` 的“开发库清库重建流程”。

## 测试环境启动

测试环境应使用 MySQL 并显式执行迁移：

```bash
set -a
source deploy/config/paas-server.env.example
set +a
export PAAS_AUTO_MIGRATE=true
export PAAS_HTTP_ADDR=:8080
go run ./cmd/paas-server
```

部署流程建议：

1. 从测试环境凭据系统注入 `MYSQL_PASSWORD`，不要提交实际 `.env`。
2. 启动前验证 MySQL 连接和字符集。
3. 启动 `paas-server`，由 `PAAS_AUTO_MIGRATE=true` 执行迁移。
4. 调用 `/healthz` 确认控制面可用。
5. 启动 Web Console，并设置 `VITE_API_BASE_URL` 指向测试环境控制面地址。

测试环境按数据保留要求分为两类：

- 临时开发验收环境：允许按开发阶段策略清库重建，用当前 migration 全量建表。
- 持久试点或生产类环境：禁止清库重建，禁止修改已执行 migration；表结构变化必须追加更高版本 migration。

MySQL 验证命令：

```bash
mysql -h"$MYSQL_HOST" -P"$MYSQL_PORT" -u"$MYSQL_USER" -p"$MYSQL_PASSWORD" "$MYSQL_DATABASE" -e \
  "SELECT VERSION(), @@character_set_server, @@collation_server;"
```

## paas-agent 部署

Agent 配置样例位于 `deploy/config/paas-agent.env.example`。测试环境应将 `PAAS_AGENT_TOKEN` 通过 Kubernetes Secret 注入，RBAC 草案位于 `deploy/paas-agent/rbac.yaml`。

集群侧最小步骤：

```bash
kubectl create namespace paas-system
kubectl apply -f deploy/paas-agent/rbac.yaml
kubectl -n paas-system create secret generic paas-agent-secret \
  --from-literal=PAAS_AGENT_TOKEN='replace-with-control-plane-issued-token'
```

当前仓库尚未提供 `paas-agent` Deployment manifest。接入测试环境时，需要基于镜像构建流程补充 Deployment，并注入以下环境变量：

- `PAAS_CLUSTER_ID`
- `PAAS_CONTROL_PLANE_URL`
- `PAAS_AGENT_TOKEN`
- `PAAS_AGENT_NAMESPACES`
- `PAAS_HEARTBEAT_INTERVAL`
- `PAAS_SNAPSHOT_INTERVAL`

Agent 只负责状态采集、事件上报和受控任务执行，不保存平台用户凭据，不直接做发布决策。

## 已知限制

- `paas-server` 业务仓储固定使用 MySQL 正式表。内存仓储和 `repository_snapshots` 快照式持久化已移除；历史快照表会在迁移中删除，旧快照数据不迁移。
- SourceRepository 已支持通过 `GITLAB_BASE_URL`、`GITLAB_TOKEN` 和 `GITLAB_ROOT_GROUP_PATH` 接入真实 GitLab；`GITLAB_WEBHOOK_CALLBACK_URL` 留空时跳过 Webhook 配置，本地开发不要配置成 `127.0.0.1`。
- Jenkins 已支持通过 `JENKINS_BASE_URL`、`JENKINS_USERNAME`、`JENKINS_TOKEN`、`JENKINS_CALLBACK_BASE_URL` 和 `IMAGE_REPOSITORY` 接入真实 Jenkins；平台管理员需先在 Web Console 上传并启用 `job.xml` 模板。
- Registry 和 Argo CD 的真实连接参数仍待接入 `paas-server` 启动环境变量。
- 当前仓库提供 Agent RBAC 草案，但尚未提供 Agent Deployment manifest 和镜像构建脚本。
