# 本地和测试环境部署说明

本文档用于把当前测试可用版启动为可本地调试、可测试环境验收的版本。配置样例见 `doc/配置样例.md`，MySQL 迁移细节见 `doc/MySQL迁移与启动流程.md`。

## 运行前提

- Go `1.24`。
- Node.js 和 npm，前端工程位于 `web/console`。
- 测试环境需要 MySQL 8.0；本地仅查看内存演示数据时可以不启用 MySQL 迁移。
- 每个 Kubernetes 集群内部署独立 Argo CD 和 `paas-agent`，普通用户不直接访问 Argo CD UI、Jenkins UI、部署清单仓库或 kubeconfig。

## 本地开发启动

本地启动默认使用内存仓储和演示数据，适合验证 API、Web Console 和中文用户流程。推荐使用一键脚本同时启动后端和前端：

```bash
cp deploy/config/paas-dev.env.example deploy/config/paas-dev.env
# 按需修改 deploy/config/paas-dev.env 中的 GitLab、Jenkins、MySQL 和镜像仓库配置
./scripts/dev-up.sh
```

脚本会读取 `deploy/config/paas-dev.env`，启动 `paas-server` 和 Web Console，并自动设置 `VITE_API_BASE_URL`。未配置 `GITLAB_BASE_URL/GITLAB_TOKEN` 或 `JENKINS_BASE_URL/JENKINS_TOKEN` 时，对应集成使用 fake adapter。

也可以手动启动后端：

```bash
set -a
source deploy/config/paas-server.env.example
set +a
export PAAS_AUTO_MIGRATE=false
export PAAS_HTTP_ADDR=:8080
go run ./cmd/paas-server
```

健康检查：

```bash
curl -fsS http://127.0.0.1:8080/healthz
```

手动启动 Web Console：

```bash
cd web/console
npm install
VITE_API_BASE_URL=http://127.0.0.1:8080 npm run dev
```

未设置 `VITE_API_BASE_URL` 时，Web Console 使用前端 mock API；设置后会调用真实后端开发 API。

本地演示账号：

```text
username = admin
password = password
```

## 测试环境启动

测试环境应使用 MySQL 并显式执行迁移：

```bash
set -a
source deploy/config/paas-server.env.example
set +a
export PAAS_AUTO_MIGRATE=true
export PAAS_HTTP_ADDR=:8080
go run ./cmd/paas-server
```

部署流程建议：

1. 从测试环境凭据系统注入 `MYSQL_PASSWORD`，不要提交实际 `.env`。
2. 启动前验证 MySQL 连接和字符集。
3. 启动 `paas-server`，由 `PAAS_AUTO_MIGRATE=true` 执行迁移。
4. 调用 `/healthz` 确认控制面可用。
5. 启动 Web Console，并设置 `VITE_API_BASE_URL` 指向测试环境控制面地址。

MySQL 验证命令：

```bash
mysql -h"$MYSQL_HOST" -P"$MYSQL_PORT" -u"$MYSQL_USER" -p"$MYSQL_PASSWORD" "$MYSQL_DATABASE" -e \
  "SELECT VERSION(), @@character_set_server, @@collation_server;"
```

## paas-agent 部署

Agent 配置样例位于 `deploy/config/paas-agent.env.example`。测试环境应将 `PAAS_AGENT_TOKEN` 通过 Kubernetes Secret 注入，RBAC 草案位于 `deploy/paas-agent/rbac.yaml`。

集群侧最小步骤：

```bash
kubectl create namespace paas-system
kubectl apply -f deploy/paas-agent/rbac.yaml
kubectl -n paas-system create secret generic paas-agent-secret \
  --from-literal=PAAS_AGENT_TOKEN='replace-with-control-plane-issued-token'
```

当前仓库尚未提供 `paas-agent` Deployment manifest。接入测试环境时，需要基于镜像构建流程补充 Deployment，并注入以下环境变量：

- `PAAS_CLUSTER_ID`
- `PAAS_CONTROL_PLANE_URL`
- `PAAS_AGENT_TOKEN`
- `PAAS_AGENT_NAMESPACES`
- `PAAS_HEARTBEAT_INTERVAL`
- `PAAS_SNAPSHOT_INTERVAL`

Agent 只负责状态采集、事件上报和受控任务执行，不保存平台用户凭据，不直接做发布决策。

## 验证命令

后端：

```bash
go test ./...
```

前端：

```bash
cd web/console
npm test -- --coverage
npm run e2e
npm run build
```

端到端人工验收应继续使用 `doc/验收清单.md`，重点确认构建、发布、回滚、审计、安全边界和失败状态展示。

## 已知限制

- `paas-server` 已支持 `PAAS_REPOSITORY_DRIVER=mysql` 切换业务仓储。当前 MySQL 业务仓储为第一阶段快照式持久化，数据保存到 `repository_snapshots`，后续可逐模块替换为逐表 SQL repository。
- SourceRepository 已支持通过 `GITLAB_BASE_URL`、`GITLAB_TOKEN` 和 `GITLAB_ROOT_GROUP_PATH` 接入真实 GitLab；`GITLAB_WEBHOOK_CALLBACK_URL` 留空时跳过 Webhook 配置，本地开发不要配置成 `127.0.0.1`。
- Jenkins 已支持通过 `JENKINS_BASE_URL`、`JENKINS_USERNAME`、`JENKINS_TOKEN`、`JENKINS_CALLBACK_BASE_URL` 和 `IMAGE_REPOSITORY` 接入真实 Jenkins；平台管理员需先在 Web Console 上传并启用 `job.xml` 模板。
- Registry 和 Argo CD 的真实连接参数仍待接入 `paas-server` 启动环境变量。
- 当前仓库提供 Agent RBAC 草案，但尚未提供 Agent Deployment manifest 和镜像构建脚本。
